SECTION 1: JS FUNDAMENTALS AND SYNTAX

Q1: What is JavaScript? What are its key characteristics?
────────────────────────────────────────────────────────────
A: JavaScript is a high-level, interpreted (JIT-compiled), dynamically typed,
   single-threaded, prototype-based programming language. Originally created to
   add interactivity to web pages, it now runs in browsers, servers (Node.js),
   mobile apps, and embedded devices.

   Key characteristics:
   • Dynamically typed  — variable types are determined at runtime, not declared.
   • Interpreted / JIT  — modern engines (V8, SpiderMonkey) compile JS to machine
                          code just-in-time for performance.
   • Single-threaded    — one call stack, executes one operation at a time, but
                          uses an event loop for concurrency (non-blocking I/O).
   • Prototype-based     — objects inherit directly from other objects via a
                          prototype chain (no classical class-based inheritance
                          at the engine level, though `class` syntax sugar exists).
   • First-class functions — functions are values: can be assigned, passed as
                          arguments, returned from other functions.
   • Multi-paradigm      — supports procedural, object-oriented, and functional
                          styles.

   JavaScript engines implement the ECMAScript (ES) specification. "JavaScript"
   is the language; ECMAScript is the standard it conforms to (ES5, ES6/ES2015,
   ES2016... yearly releases since ES6).

────────────────────────────────────────────────────────────────────────────────
Q2: What is the difference between var, let, and const?
──────────────────────────────────────────────────────────
A: All three declare variables, but differ in scope, hoisting behavior, and
   mutability.

   var:
   • Function-scoped (or global-scoped if outside a function) — NOT block-scoped.
   • Hoisted and initialized with `undefined` at the top of its scope.
   • Can be re-declared and re-assigned.
   • Creates a property on the global object (`window`/`globalThis`) when
     declared at the top level in non-module scripts.

   let:
   • Block-scoped ({ } defines its scope — if, for, while, plain blocks).
   • Hoisted but NOT initialized — accessing before declaration throws a
     ReferenceError (Temporal Dead Zone, see Q6).
   • Can be re-assigned but not re-declared in the same scope.

   const:
   • Block-scoped, same TDZ behavior as let.
   • Must be initialized at declaration.
   • Cannot be re-assigned — but if the value is an object/array, its CONTENTS
     can still be mutated (the binding is constant, not the value).

   Example:
     function demo() {
       if (true) {
         var  a = 1;   // function-scoped — leaks outside the if block
         let  b = 2;   // block-scoped — only visible inside the if block
         const c = 3;  // block-scoped
       }
       console.log(a);        // 1
       console.log(b); // error
       console.log(c); // error
     }

     const arr = [1, 2, 3];
     arr.push(4);       // OK — mutating contents
     arr = [5, 6];       // TypeError: Assignment to constant variable.

   Rule of thumb: prefer `const` by default, use `let` when reassignment is
   needed, avoid `var` in modern code.

────────────────────────────────────────────────────────────────────────────────
Q3: What are the data types in JavaScript? Primitive vs Reference types?
───────────────────────────────────────────────────────────────────────────
A: JavaScript has 8 data types — 7 primitives and 1 reference (object) type.

   PRIMITIVE TYPES (immutable, compared/copied by VALUE):
   • string    — "hello", 'hello', `hello`
   • number    — 42, 3.14, NaN, Infinity (IEEE-754 double precision)
   • bigint    — 123456789012345678901234567890n (arbitrary precision integers)
   • boolean   — true, false
   • undefined — a variable declared but not assigned
   • null      — intentional absence of value
   • symbol    — unique, immutable identifier (ES6), often used as object keys

   REFERENCE TYPE:
   • object (includes: plain objects, arrays, functions, Date, RegExp, Map,
     Set, etc.) — stored on the heap, variable holds a REFERENCE (pointer).

   Primitive vs Reference behavior:
     let a = 10;
     let b = a;      // copies the VALUE
     b = 20;
     console.log(a); // 10 — unaffected
     console.log(b); // 20

     let obj1 = { x: 1 };
     let obj2 = obj1; // copies the REFERENCE
     obj2.x = 99;
     console.log(obj1.x); // 99 — same underlying object

   typeof quirks:
     typeof "hi"        // "string"
     typeof 42          // "number"
     typeof true         // "boolean"
     typeof undefined    // "undefined"
     typeof Symbol()     // "symbol"
     typeof 10n           // "bigint"
     typeof null         // "object"   ← famous historical bug, never fixed (backward compat)
     typeof {}           // "object"
     typeof []           // "object"   ← use Array.isArray() to detect arrays
     typeof function(){} // "function" ← functions are callable objects

────────────────────────────────────────────────────────────────────────────────
Q4: What is dynamic typing, and how does it differ from weak/strong typing?
────────────────────────────────────────────────────────────────────────────
A: DYNAMIC TYPING means variable types are determined at RUNTIME, not declared
   ahead of time — the same variable can hold different types over its lifetime:
     let x = 5;        // x is a number
     x = "hello";       // now x is a string — legal, no error

   This is different from STATIC TYPING (TypeScript, Java, C#) where a
   variable's type is fixed at compile time.

   JavaScript is also WEAKLY (loosely) TYPED — it performs implicit type
   coercion between types during operations:
     "5" + 3    // "53"  (number coerced to string)
     "5" - 3    // 2     (string coerced to number)
     true + 1   // 2     (boolean coerced to number)

   A STRONGLY typed dynamic language (e.g., Python) would throw an error on
   mismatched-type operations instead of silently coercing. JavaScript's
   combination of dynamic + weak typing is a common source of bugs, which is
   why TypeScript (static typing on top of JS) and strict equality (===) are
   widely recommended.

────────────────────────────────────────────────────────────────────────────────
Q5: What are template literals? What is strict mode?
────────────────────────────────────────────────────────
A: TEMPLATE LITERALS (ES6) — strings delimited by backticks (`) that support:
   • String interpolation: ${expression}
   • Multi-line strings without \n concatenation
   • Tagged templates (custom processing function before the string is built)

     const name = "Alice";
     const age = 30;
     const msg = `Hello ${name}, you are ${age} years old.`;
     const multi = `Line 1
     Line 2`;               // preserves the newline literally

     function tag(strings, ...values) {
       return strings.reduce((acc, s, i) => acc + s + (values[i] ?? ""), "");
     }
     tag`Name: ${name}, Age: ${age}`;   // tagged template

   STRICT MODE — `"use strict";` at the top of a file or function opts into a
   restricted variant of JS that catches common mistakes and disables unsafe
   features:
   • Assigning to an undeclared variable throws a ReferenceError (instead of
     silently creating a global).
   • `this` is `undefined` in a plain function call (instead of the global object).
   • Duplicate parameter names are a SyntaxError.
   • Assigning to a read-only/non-writable property throws instead of failing
     silently.
   • Octal literals (010) are disallowed.
   • `with` statement is disallowed.

     "use strict";
     x = 10;  // ReferenceError: x is not defined

   ES6 modules and classes are automatically in strict mode — no directive
   needed.

SECTION 2: SCOPE, HOISTING AND CLOSURES
================================================================================

Q6: What is hoisting? How do var, let, const, and function declarations differ?
────────────────────────────────────────────────────────────────────────────────
A: Hoisting is JS engine behavior where declarations are processed before code
   execution — during the "creation phase" of an execution context, memory is
   allocated for variables and functions before any line actually runs.

   var:
   • Hoisted AND initialized to `undefined`.
     console.log(a); // undefined  (not an error)
     var a = 5;

   let / const:
   • Hoisted but NOT initialized. They exist in a "Temporal Dead Zone" (TDZ)
     from the start of the block until the declaration line executes.
     console.log(b); // ReferenceError: Cannot access 'b' before initialization
     let b = 5;

   function declarations:
   • Fully hoisted — both the name AND the function body/definition are
     hoisted, so you can call it before its textual position.
     greet();               // "hi" — works
     function greet() { console.log("hi"); }

   function expressions (including arrow functions) assigned to var/let/const:
   • Only the variable is hoisted per its declaration keyword's rules — the
     function VALUE is not available until the assignment line executes.
     sayHi();                 // TypeError: sayHi is not a function (var → undefined)
     var sayHi = function() { console.log("hi"); };

   class declarations:
   • Hoisted like let/const — in the TDZ until evaluated. Cannot be used
     before declaration.

────────────────────────────────────────────────────────────────────────────────
Q7: What is the Temporal Dead Zone (TDZ)?
─────────────────────────────────────────────
A: The TDZ is the span of code between the start of a block scope and the
   point where a let/const/class declaration is actually executed. Accessing
   the variable in this zone throws a ReferenceError, even though the
   variable has technically been "hoisted" (its binding exists, but is
   uninitialized).

     {
       // TDZ for `x` starts here
       console.log(x); // ReferenceError: Cannot access 'x' before initialization
       let x = 10;      // TDZ ends here
     }

   The TDZ exists to catch bugs early — with `var`, using a variable before
   its declaration silently gives `undefined`, masking mistakes. TDZ makes
   this a hard error instead.

   Note: `typeof` on a TDZ variable also throws (unlike a truly undeclared
   variable, where typeof safely returns "undefined"):
     console.log(typeof y); // ReferenceError (y is in TDZ)
     let y = 1;

     console.log(typeof z); // "undefined" — z was never declared at all, no TDZ

────────────────────────────────────────────────────────────────────────────────
Q8: What is the difference between function scope and block scope?
────────────────────────────────────────────────────────────────────
A: FUNCTION SCOPE — a variable is visible throughout the entire function it
   was declared in, regardless of nested blocks (if/for/while). `var` uses
   function scope.

     function fn() {
       if (true) {
         var x = 1;
       }
       console.log(x); // 1 — accessible outside the if block
     }

   BLOCK SCOPE — a variable is visible only within the nearest enclosing pair
   of curly braces { }. `let` and `const` use block scope.

     function fn() {
       if (true) {
         let y = 1;
       }
       console.log(y); // ReferenceError: y is not defined
     }

   Block scope applies to any { }: if, for, while, switch, or even a bare
   { } block. This is why `let` in a for-loop creates a NEW binding per
   iteration (see Q9 closures-in-loops example).

────────────────────────────────────────────────────────────────────────────────
Q9: What is a closure? Give a practical example.
────────────────────────────────────────────────────
A: A closure is formed when a function "remembers" the lexical scope it was
   defined in, even after the outer function has returned. The inner function
   retains access to the outer function's variables.

     function makeCounter() {
       let count = 0;              // enclosed variable
       return function () {
         count++;                  // closure over `count`
         return count;
       };
     }
     const counter = makeCounter();
     counter(); // 1
     counter(); // 2
     counter(); // 3 — `count` persists between calls, private to this instance

   Each call to makeCounter() creates a NEW closure with its own independent
   `count`:
     const counter2 = makeCounter();
     counter2(); // 1 — independent from `counter`

   Practical uses of closures:
   • Data privacy / encapsulation (module pattern — see Section 14).
   • Function factories (createMultiplier(2), createMultiplier(3)).
   • Memoization / caching.
   • Callbacks and event handlers that need access to outer state.
   • Currying and partial application.

   Classic loop pitfall (var vs let):
     for (var i = 0; i < 3; i++) {
       setTimeout(() => console.log(i), 0);
     }
     // prints: 3, 3, 3  — all closures share the SAME `i` (function-scoped var);
     // by the time the callbacks run, the loop has finished and i === 3.

     for (let i = 0; i < 3; i++) {
       setTimeout(() => console.log(i), 0);
     }
     // prints: 0, 1, 2 — `let` creates a NEW binding of `i` per iteration,
     // so each closure captures its own snapshot.

────────────────────────────────────────────────────────────────────────────────
Q10: What is an IIFE (Immediately Invoked Function Expression)?
────────────────────────────────────────────────────────────────
A: An IIFE is a function defined and executed immediately, used to create a
   private scope and avoid polluting the global namespace.

     (function () {
       var privateVar = "I'm private";
       console.log(privateVar);
     })();                          // parentheses invoke it immediately

     (() => {
       console.log("arrow IIFE");
     })();

     // Common historical use: module pattern before ES6 modules existed.
     const myModule = (function () {
       let counter = 0;
       return {
         increment: () => ++counter,
         getCount: () => counter
       };
     })();

   Why the outer parentheses? `function () {}` alone is a function
   DECLARATION and requires a name; wrapping it in ( ) turns it into a
   function EXPRESSION, which can then be invoked immediately with a
   trailing ().

   With ES6 block scope (let/const) and native modules, the need for IIFEs
   has diminished, but they're still used for isolated one-off scripts,
   avoiding global leakage in non-module <script> tags.

────────────────────────────────────────────────────────────────────────────────
Q11: What is lexical scoping?
─────────────────────────────
A: Lexical (static) scoping means a variable's scope is determined by WHERE it
   is written in the source code — not by how/where the function is called
   from. JavaScript resolves variable references by walking up the chain of
   enclosing scopes AS WRITTEN, not the call stack.

     let x = "global";
     function outer() {
       let x = "outer";
       function inner() {
         console.log(x); // "outer" — resolved lexically to the enclosing
                          // `outer` function's scope, regardless of where
                          // inner() is eventually called from.
       }
       return inner;
     }
     outer()();  // "outer"

   This is the mechanism that makes closures possible — the scope chain is
   fixed at definition time. Contrast with `this`, which is DYNAMIC (call-site
   determined) in a regular function, unlike lexically-scoped variables.

SECTION 3: EQUALITY, COERCION AND TRUTHINESS
================================================================================

Q12: What is the difference between == and ===?
──────────────────────────────────────────────────
A: === (strict equality) — compares value AND type. No coercion. Two values
   are equal only if they are the same type and same value.

   == (loose/abstract equality) — coerces operands to the same type before
   comparing, following the Abstract Equality Comparison algorithm.

     5 === "5"     // false — different types
     5 == "5"      // true  — "5" coerced to number 5

     null === undefined  // false — different types
     null == undefined   // true  — special-cased: they are only == to each other and themselves

     NaN === NaN    // false — NaN is never equal to anything, including itself
     0 === -0       // true  — === treats +0 and -0 as equal (see Object.is, Q14)

   Best practice: always use === / !== unless you have a specific, well-
   understood reason to coerce (e.g., `if (x == null)` is a common idiom that
   checks for both null AND undefined in one comparison).

────────────────────────────────────────────────────────────────────────────────
Q13: What are the rules for type coercion in JavaScript?
─────────────────────────────────────────────────────────
A: Coercion happens implicitly (via operators like ==, +, if conditions) or
   explicitly (String(x), Number(x), Boolean(x)).

   THE + OPERATOR:
   • If either operand is a string, both are converted to strings and
     concatenated.
   • Otherwise, both are converted to numbers and added.
     1 + "2"      // "12"  (number → string)
     1 + 2 + "3"  // "33"  (1+2=3, then 3+"3"="33" — left to right)
     "3" + 1 + 2  // "312" (string first — all subsequent + are string concat)
     1 + true     // 2     (true → 1)
     1 + null     // 1     (null → 0)
     1 + undefined// NaN   (undefined → NaN)
     [] + []      // ""    (both arrays → "")
     [] + {}      // "[object Object]" (both → strings, concatenated)

   OTHER ARITHMETIC OPERATORS (-, *, /, %):
   • Always coerce both operands to numbers.
     "5" - 2      // 3
     "5" * "2"    // 10
     true - 1     // 0

   == (loose equality) coercion rules (simplified):
   • null == undefined → true (special case, nothing else equals either).
   • number == string  → string converted to number.
   • boolean == anything → boolean converted to number first, then compared.
   • object == primitive → object converted to primitive via ToPrimitive
     (calls valueOf()/toString()).
     [1,2] == "1,2"   // true — array's toString() → "1,2"

   Object → primitive conversion (ToPrimitive) order:
   • For "number" hint: tries valueOf() first, then toString().
   • For "string" hint: tries toString() first, then valueOf().
   • Symbol.toPrimitive, if defined, takes priority over both.

────────────────────────────────────────────────────────────────────────────────
Q14: What are truthy and falsy values? What is Object.is()?
────────────────────────────────────────────────────────────
A: FALSY values (coerce to `false` in a boolean context) — there are exactly
   8 in JavaScript:
     false, 0, -0, 0n (BigInt zero), "", null, undefined, NaN

   Everything else is TRUTHY, including:
     "0" (non-empty string), "false" (non-empty string), [] (empty array),
     {} (empty object), function(){}, Infinity, -Infinity

     if ([]) console.log("truthy!");   // logs — empty array is truthy
     if ("0") console.log("truthy!");  // logs — non-empty string is truthy

   Object.is(a, b) — similar to === but with two differences, used for
   edge-case-safe equality:
     Object.is(NaN, NaN)   // true   (=== gives false)
     Object.is(0, -0)      // false  (=== gives true)
     Object.is(5, 5)       // true   (same as ===)
     Object.is({}, {})     // false  (different references, same as ===)

   Object.is is used internally by React (in useState bailout checks) and
   useful for precise value comparisons, but is NOT a general replacement
   for === in everyday code.

────────────────────────────────────────────────────────────────────────────────
Q15: Why does NaN behave strangely? How do you correctly check for it?
─────────────────────────────────────────────────────────────────────────
A: NaN ("Not a Number") is a special numeric value representing an invalid or
   unrepresentable number result (e.g., 0/0, parseInt("abc"), Math.sqrt(-1)).

   Quirks:
   • typeof NaN === "number"  — NaN is still of type "number".
   • NaN !== NaN, and NaN == NaN is also false — by IEEE-754 spec, NaN is
     defined to be unequal to everything, including itself. This is the ONLY
     value in JS that is not equal to itself.

   Correct ways to check for NaN:
     Number.isNaN(x)      // ✅ correct — true only if x IS the NaN value
     isNaN(x)              // ⚠️ coerces x to a number first — isNaN("foo") is
                            //    true because "foo" → NaN, even though "foo"
                            //    itself isn't the NaN value.
     Object.is(x, NaN)     // ✅ also correct
     x !== x                // ✅ classic trick — only true for NaN

     Number.isNaN("foo")   // false — "foo" is not the NaN value (no coercion)
     isNaN("foo")           // true  — "foo" coerced to NaN first (misleading)

Q16: What is the difference between function declarations, function
     expressions, and arrow functions?
──────────────────────────────────────────────────────────────────────
A: FUNCTION DECLARATION:
     function add(a, b) { return a + b; }
   • Hoisted completely (name + body) — callable before its position in code.
   • Has its own `this`, `arguments`, and can be used as a constructor (`new`).

   FUNCTION EXPRESSION:
     const add = function (a, b) { return a + b; };
     const add2 = function namedAdd(a, b) { return a + b; }; // named expr
   • Not hoisted with its definition (only the variable binding is, per
     var/let/const rules).
   • Has its own `this`, `arguments`, usable as a constructor.

   ARROW FUNCTION (ES6):
     const add = (a, b) => a + b;
   • Concise syntax, implicit return for single expressions.
   • Does NOT have its own `this` — inherits `this` lexically from the
     enclosing scope (see Section 5).
   • Does NOT have its own `arguments` object — inherits from enclosing scope
     (use rest parameters `...args` instead).
   • Cannot be used as a constructor — `new (() => {})()` throws TypeError.
   • No `prototype` property.
   • Cannot be used as a generator (no `function*` arrow equivalent).

   Practical implication — `this` binding is the #1 reason to choose one over
   the other:
     const obj = {
       name: "Widget",
       regularMethod: function () {
         setTimeout(function () {
           console.log(this.name); // undefined — `this` is not `obj` here
                                    // (plain function call, `this` = global/undefined)
         }, 100);
       },
       arrowMethod: function () {
         setTimeout(() => {
           console.log(this.name); // "Widget" — arrow inherits `this` from
                                    // arrowMethod's scope, which is `obj`
         }, 100);
       }
     };

────────────────────────────────────────────────────────────────────────────────
Q17: What are default parameters and rest parameters?
──────────────────────────────────────────────────────
A: DEFAULT PARAMETERS (ES6) — provide a fallback value when an argument is
   `undefined` (not passed, or explicitly passed as undefined):
     function greet(name = "Guest", greeting = "Hello") {
       return `${greeting}, ${name}!`;
     }
     greet();               // "Hello, Guest!"
     greet("Alice");        // "Hello, Alice!"
     greet(undefined, "Hi");// "Hi, Guest!" — undefined triggers default
     greet(null);            // "Hello, null!" — null does NOT trigger default

   Defaults can reference earlier parameters:
     function box(width, height = width) { return width * height; }
     box(5); // 25

   REST PARAMETERS (ES6) — collects remaining arguments into a real array,
   must be the LAST parameter:
     function sum(...numbers) {
       return numbers.reduce((acc, n) => acc + n, 0);
     }
     sum(1, 2, 3, 4); // 10

     function logFirst(first, ...rest) {
       console.log(first, rest);
     }
     logFirst(1, 2, 3); // 1 [2, 3]

   Rest vs the old `arguments` object:
   • `arguments` is array-LIKE (has length/index access) but not a real
     array — no .map()/.filter()/.reduce() without converting it first.
   • Rest parameters produce a real Array immediately.
   • Arrow functions have no `arguments` at all — rest parameters are the
     only option there.

────────────────────────────────────────────────────────────────────────────────
Q18: What are higher-order functions? Give examples.
────────────────────────────────────────────────────
A: A higher-order function (HOF) either:
   • Takes one or more functions as arguments, and/or
   • Returns a function as its result.

   This is possible because functions are "first-class citizens" in JS —
   treated like any other value.

     // Takes a function as an argument
     [1, 2, 3].map(x => x * 2);              // [2, 4, 6]
     [1, 2, 3].filter(x => x % 2 === 0);      // [2]
     [1, 2, 3].reduce((acc, x) => acc + x, 0);// 6

     // Returns a function
     function multiplyBy(factor) {
       return function (n) { return n * factor; };
     }
     const double = multiplyBy(2);
     double(5); // 10

   Built-in HOFs: map, filter, reduce, forEach, sort (with comparator),
   setTimeout, addEventListener (takes a callback), Array.from (with a map
   function), Promise.then.

────────────────────────────────────────────────────────────────────────────────
Q19: What are currying and partial application? How do you implement them?
─────────────────────────────────────────────────────────────────────────────
A: CURRYING — transforming a function that takes multiple arguments into a
   sequence of functions, each taking a SINGLE argument.

     function curry(fn) {
       return function curried(...args) {
         if (args.length >= fn.length) {
           return fn.apply(this, args);
         }
         return (...moreArgs) => curried(...args, ...moreArgs);
       };
     }

     function add3(a, b, c) { return a + b + c; }
     const curriedAdd = curry(add3);

     curriedAdd(1)(2)(3);     // 6
     curriedAdd(1, 2)(3);     // 6 — can still pass multiple at once
     curriedAdd(1)(2, 3);     // 6
     curriedAdd(1, 2, 3);     // 6

   PARTIAL APPLICATION — fixing (pre-filling) some arguments of a function,
   producing a new function that takes the remaining arguments. Similar to
   currying but not restricted to one argument at a time.

     function partial(fn, ...presetArgs) {
       return (...laterArgs) => fn(...presetArgs, ...laterArgs);
     }

     function greet(greeting, name) { return `${greeting}, ${name}!`; }
     const sayHello = partial(greet, "Hello");
     sayHello("Alice"); // "Hello, Alice!"

   Difference: currying always breaks a function into UNARY (single-argument)
   steps and is typically automatic/recursive; partial application manually
   fixes an arbitrary number of arguments in one step. Both improve
   reusability and enable point-free/functional composition style.

Q20: What is memoization? Implement a simple memoize function.
─────────────────────────────────────────────────────────────────
A: Memoization is a caching optimization: store the results of expensive
   function calls keyed by their arguments, and return the cached result
   when the same arguments occur again — trading memory for speed. Works
   best on PURE functions (same input always produces same output, no side
   effects).

     function memoize(fn) {
       const cache = new Map();
       return function (...args) {
         const key = JSON.stringify(args);
         if (cache.has(key)) {
           return cache.get(key);
         }
         const result = fn.apply(this, args);
         cache.set(key, result);
         return result;
       };
     }

     function slowSquare(n) {
       for (let i = 0; i < 1e8; i++); // simulate expensive work
       return n * n;
     }
     const fastSquare = memoize(slowSquare);
     fastSquare(5); // slow the first time
     fastSquare(5); // instant — served from cache

   Classic use case — memoized Fibonacci:
     const fib = memoize(function (n) {
       return n <= 1 ? n : fib(n - 1) + fib(n - 2);
     });
     fib(40); // fast — without memoization this is exponential time

   Caveats: JSON.stringify as a cache key fails for functions, symbols,
   circular references, and doesn't distinguish object identity — for
   production use, consider WeakMap-based caches for object arguments (helps
   garbage collection too, see Section 12).

────────────────────────────────────────────────────────────────────────────────
Q21: What is a pure function? Why does it matter?
───────────────────────────────────────────────────
A: A pure function:
   1. Given the same input, ALWAYS returns the same output (deterministic).
   2. Has NO side effects — does not mutate external state, does not perform
      I/O, does not modify its arguments.

     // Pure
     function add(a, b) { return a + b; }

     // Impure — depends on external state
     let tax = 0.1;
     function addTax(price) { return price + price * tax; } // tax can change

     // Impure — mutates argument (side effect)
     function addItem(cart, item) {
       cart.push(item); // mutates the caller's array
       return cart;
     }
     // Pure version:
     function addItemPure(cart, item) {
       return [...cart, item]; // returns a new array
     }

   Why it matters:
   • Predictable & testable — no hidden dependencies or mocking needed.
   • Safe to memoize/cache.
   • Safe for concurrent/parallel execution — no shared mutable state.
   • Easier to reason about and debug.
   • Foundational to functional programming, React's rendering model, and
     Redux reducers (which must be pure).

────────────────────────────────────────────────────────────────────────────────
Q22: What is the difference between call(), apply(), and bind()?
───────────────────────────────────────────────────────────────────
A: All three explicitly set the `this` value for a function call, overriding
   the default binding rules.

   call(thisArg, arg1, arg2, ...) — invokes the function IMMEDIATELY with
   arguments passed individually (comma-separated).

     function greet(greeting) { return `${greeting}, ${this.name}`; }
     const user = { name: "Alice" };
     greet.call(user, "Hello"); // "Hello, Alice" — invoked immediately

   apply(thisArg, [argsArray]) — invokes the function IMMEDIATELY with
   arguments passed as an ARRAY.

     greet.apply(user, ["Hi"]); // "Hi, Alice"
     // Classic pre-spread use: Math.max.apply(null, [1, 5, 3]) === 5

   bind(thisArg, arg1, arg2, ...) — does NOT invoke immediately; instead
   returns a NEW function with `this` permanently bound (and optionally
   some leading arguments pre-filled — partial application).

     const greetAlice = greet.bind(user, "Hey");
     greetAlice(); // "Hey, Alice" — called later, `this` is locked to `user`

   Common practical use — fixing `this` in event handlers/callbacks before
   arrow functions were widely used:
     class Counter {
       constructor() {
         this.count = 0;
         this.increment = this.increment.bind(this); // lock `this`
       }
       increment() { this.count++; }
     }

   With modern arrow functions and class fields, explicit binding is needed
   less often, but call/apply/bind remain essential for function borrowing
   (e.g., using Array.prototype methods on array-like objects) and for
   implementing currying/partial-application utilities.

================================================================================
SECTION 5: THE `this` KEYWORD AND EXECUTION CONTEXT
================================================================================

Q23: How is `this` determined in JavaScript? Explain the different binding rules.
─────────────────────────────────────────────────────────────────────────────────
A: Unlike lexically-scoped variables, `this` is determined DYNAMICALLY by HOW
   a function is CALLED (the call-site), not where it's defined — except for
   arrow functions. There are four main binding rules, in order of precedence
   (highest first):

   1. NEW BINDING — calling a function with `new` creates a brand new object,
      and `this` is bound to it.
        function Person(name) { this.name = name; }
        const p = new Person("Alice"); // this = the new object

   2. EXPLICIT BINDING — call(), apply(), bind() explicitly set `this`.
        greet.call(someObject);

   3. IMPLICIT BINDING — calling a function as a METHOD of an object; `this`
      is the object the method was called ON (the object left of the dot at
      the call-site).
        const obj = { name: "Bob", greet() { return this.name; } };
        obj.greet(); // this === obj

      Pitfall — losing implicit binding when a method is detached:
        const fn = obj.greet;
        fn(); // this is undefined (strict mode) or global object (sloppy mode)
              // — NOT obj, because the call-site is now a plain function call

   4. DEFAULT BINDING — a plain function call with no object context.
      • Strict mode: `this` is `undefined`.
      • Non-strict (sloppy) mode: `this` is the global object (`window` in
        browsers, `global` in Node — technically `globalThis`).
        function standalone() { return this; }
        standalone(); // undefined (strict) or globalThis (sloppy)

   5. ARROW FUNCTIONS — ignore all of the above. Arrow functions have NO own
      `this`; they capture `this` LEXICALLY from their enclosing scope at
      definition time, and it can never be changed by call/apply/bind/new.
        const obj = {
          name: "Carol",
          greet: () => this.name // `this` here is NOT obj — it's whatever
                                  // `this` was in the surrounding scope
                                  // (e.g., module scope / undefined)
        };

   Precedence order when multiple rules could apply: new > explicit
   (call/apply/bind, and bind cannot be overridden by a later call/apply) >
   implicit (method call) > default.

────────────────────────────────────────────────────────────────────────────────
Q24: What is the call stack and an execution context?
──────────────────────────────────────────────────────
A: EXECUTION CONTEXT is the environment in which JS code is evaluated and
   executed. Every time a function is invoked, a new execution context is
   created. There are three kinds:
   • Global Execution Context — created once, when the script first runs.
   • Function Execution Context — created every time a function is called.
   • Eval Execution Context — created inside eval() (rarely used/discouraged).

   Each execution context has two phases:
   1. Creation phase — the `this` binding is determined, the scope chain is
      set up, and variables/functions are hoisted (memory allocated,
      var → undefined, function declarations → fully defined, let/const →
      TDZ).
   2. Execution phase — code runs line by line, assigning actual values.

   THE CALL STACK is a LIFO (last-in, first-out) stack of execution contexts
   that tracks "where the program is" — which function is currently running
   and which functions called it.

     function a() { b(); }
     function b() { c(); }
     function c() { console.log("deep"); }
     a();
     // Call stack grows:  [global] → [a] → [b] → [c]
     // c() finishes and pops off:   [global] → [a] → [b]
     // b() finishes and pops off:   [global] → [a]
     // a() finishes and pops off:   [global]

   If the stack grows without bound (e.g., infinite/unterminated recursion),
   you get a "Maximum call stack size exceeded" (RangeError) — a stack
   overflow.

   Because JS has a SINGLE call stack, it is single-threaded: only one piece
   of JS code executes at any given instant. Asynchronous behavior (timers,
   I/O, promises) is handled by the event loop feeding callbacks back onto
   this same stack one at a time (see Section 8).

────────────────────────────────────────────────────────────────────────────────
Q25: What happens if you use `this` inside a nested regular function vs an
     arrow function inside a method?
────────────────────────────────────────────────────────────────────────────
A: This is one of the most common `this`-related interview traps.

     const obj = {
       name: "Widget",
       regular() {
         function inner() {
           console.log(this); // NOT obj — plain function call → default
                               // binding → undefined (strict) / global (sloppy)
         }
         inner();
       },
       arrow() {
         const inner = () => {
           console.log(this); // obj — arrow captures `this` from `arrow()`'s
                               // scope, where `this` IS obj (method call)
         };
         inner();
       }
     };

   Pre-ES6 workaround for the regular-function case — capture `this` in a
   variable (commonly named `self` or `that`):
     regularOldStyle() {
       const self = this;
       function inner() {
         console.log(self.name); // works via closure over `self`
       }
       inner();
     }

   This is precisely the problem arrow functions were designed to solve —
   they eliminate the need for `self`/`that`/`.bind(this)` workarounds in
   nested callbacks.

SECTION 6: OBJECTS AND PROTOTYPES
================================================================================

Q26: What is prototypal inheritance? What is the prototype chain?
───────────────────────────────────────────────────────────────────
A: Every JS object has an internal, hidden link to another object called its
   PROTOTYPE (accessible via Object.getPrototypeOf(obj) or the deprecated
   __proto__ property). When you access a property on an object and it isn't
   found directly on that object, the engine looks up the prototype, then
   the prototype's prototype, and so on — this chain is the PROTOTYPE CHAIN.
   It terminates at Object.prototype, whose own prototype is `null`.

     const animal = {
       eat() { return `${this.name} is eating`; }
     };
     const dog = Object.create(animal); // dog's prototype = animal
     dog.name = "Rex";
     dog.eat(); // "Rex is eating" — eat() not on dog, found on its prototype

     Object.getPrototypeOf(dog) === animal;             // true
     Object.getPrototypeOf(animal) === Object.prototype; // true
     Object.getPrototypeOf(Object.prototype);             // null — end of chain

   Property lookup walks UP the chain; property WRITES (assignment) always
   create/set an OWN property on the object itself (does not mutate the
   prototype), a behavior called "shadowing":
     dog.eat = function () { return "Rex refuses to eat"; }; // shadows animal.eat
     dog.eat(); // "Rex refuses to eat"
     animal.eat(); // unaffected — still the original

   This is fundamentally different from classical (class-based) inheritance
   in languages like Java/C#: JS objects inherit directly from other LIVE
   OBJECTS, not from a static blueprint/class definition. `class` syntax in
   ES6 is syntactic sugar over this same prototype mechanism.

────────────────────────────────────────────────────────────────────────────────
Q27: What is the difference between Object.create() and a constructor
     function with `new`?
──────────────────────────────────────────────────────────────────────
A: CONSTRUCTOR FUNCTION + new — the traditional pre-ES6 way to create objects
   with shared behavior:
     function Person(name) {
       this.name = name;              // own property
     }
     Person.prototype.greet = function () {  // shared via prototype
       return `Hi, I'm ${this.name}`;
     };
     const p = new Person("Alice");
     p.greet(); // "Hi, I'm Alice"

   What `new Fn(...)` actually does, step by step:
   1. Creates a brand new empty object: {}.
   2. Sets the new object's prototype to Fn.prototype
      (Object.setPrototypeOf(obj, Fn.prototype)).
   3. Calls Fn with `this` bound to the new object.
   4. If Fn returns an object explicitly, that object is returned instead;
      otherwise the newly created object (`this`) is returned automatically.

   OBJECT.CREATE() — creates a new object with an EXPLICITLY specified
   prototype, no constructor function or `new` involved:
     const personProto = {
       greet() { return `Hi, I'm ${this.name}`; }
     };
     const p2 = Object.create(personProto);
     p2.name = "Bob";
     p2.greet(); // "Hi, I'm Bob"

     const nullProtoObj = Object.create(null); // no prototype at all —
                                                  // no inherited methods like
                                                  // toString/hasOwnProperty;
                                                  // useful as a pure dictionary

   Object.create gives direct, explicit control over the prototype without
   the ceremony (and gotchas, like forgetting `new`) of constructor
   functions. Modern code typically uses `class` syntax instead of either
   for most cases, but Object.create is still valuable for pure prototypal
   patterns and creating null-prototype dictionary objects.

────────────────────────────────────────────────────────────────────────────────
Q28: How do ES6 classes work? What do extends and super do?
────────────────────────────────────────────────────────────
A: ES6 `class` is syntactic sugar over the existing prototype system — it
   does not introduce a new inheritance model, just cleaner syntax.

     class Animal {
       constructor(name) {
         this.name = name;
       }
       speak() {
         return `${this.name} makes a sound.`;
       }
       static create(name) {          // static method — called on the class itself
         return new Animal(name);
       }
     }

     class Dog extends Animal {       // Dog.prototype's prototype = Animal.prototype
       constructor(name, breed) {
         super(name);                 // MUST call super() before using `this`
                                       // — calls Animal's constructor
         this.breed = breed;
       }
       speak() {
         return `${super.speak()} Woof!`; // super.method() calls the parent's version
       }
     }

     const d = new Dog("Rex", "Labrador");
     d.speak(); // "Rex makes a sound. Woof!"
     d instanceof Dog;    // true
     d instanceof Animal; // true — prototype chain

   Key facts about classes:
   • Class declarations are NOT hoisted like function declarations — they're
     in the TDZ (like let/const) — cannot be used before definition.
   • Class body code runs in strict mode automatically.
   • `extends` sets up the prototype chain: Dog.prototype's [[Prototype]] is
     Animal.prototype, AND Dog's own [[Prototype]] is Animal (for static
     inheritance).
   • In a derived class (uses `extends`), `super()` MUST be called in the
     constructor before accessing `this` — otherwise ReferenceError.
   • Methods defined in a class body are non-enumerable by default (unlike
     object literal methods, which are enumerable) — matches how built-in
     prototype methods behave.

────────────────────────────────────────────────────────────────────────────────
Q29: What are getters and setters?
──────────────────────────────────
A: Getters and setters let you define computed properties that run code on
   access/assignment, while still being used with normal property syntax
   (no function-call parentheses).

     const person = {
       firstName: "John",
       lastName: "Doe",
       get fullName() {
         return `${this.firstName} ${this.lastName}`;
       },
       set fullName(value) {
         [this.firstName, this.lastName] = value.split(" ");
       }
     };
     console.log(person.fullName);   // "John Doe" — invokes the getter
     person.fullName = "Jane Smith"; // invokes the setter
     console.log(person.firstName);  // "Jane"

   In classes:
     class Circle {
       constructor(radius) { this._radius = radius; }
       get area() { return Math.PI * this._radius ** 2; }
       set radius(value) {
         if (value <= 0) throw new RangeError("Radius must be positive");
         this._radius = value;
       }
     }
     const c = new Circle(5);
     c.area;         // 78.53... — computed on access, no () needed
     c.radius = 10;  // runs validation logic in the setter

   Use cases: computed/derived properties, validation on assignment,
   encapsulation (exposing a controlled interface over a private
   `_underscore` convention field or, in modern JS, a true `#privateField`).

────────────────────────────────────────────────────────────────────────────────
Q30: What are the important static Object methods? (freeze, seal, assign,
     keys/values/entries, defineProperty)
──────────────────────────────────────────────────────────────────────────
A: Object.freeze(obj) — makes an object IMMUTABLE (shallow):
   • Cannot add, remove, or reassign existing properties.
   • Assignment attempts fail silently in sloppy mode, throw TypeError in
     strict mode.
   • SHALLOW — nested objects inside a frozen object are still mutable.
     const o = Object.freeze({ a: 1, nested: { b: 2 } });
     o.a = 99;         // no effect (throws in strict mode)
     o.nested.b = 99;  // WORKS — nested object is not frozen
     Object.isFrozen(o); // true

   Object.seal(obj) — prevents adding/removing properties, but EXISTING
   properties can still have their VALUES changed (just not deleted or
   reconfigured):
     const o2 = Object.seal({ a: 1 });
     o2.a = 2;       // OK
     delete o2.a;    // fails — property stays
     o2.b = 3;       // fails — can't add new properties

   Object.assign(target, ...sources) — shallow-copies enumerable own
   properties from source(s) into target, returns target. Used for shallow
   cloning and merging.
     const merged = Object.assign({}, { a: 1 }, { b: 2 }); // { a: 1, b: 2 }
     // Modern equivalent: const merged2 = { ...{a:1}, ...{b:2} };

   Object.keys(obj) / Object.values(obj) / Object.entries(obj) — return
   arrays of an object's own enumerable property names / values / [key,
   value] pairs respectively:
     const o3 = { x: 1, y: 2 };
     Object.keys(o3);    // ["x", "y"]
     Object.values(o3);  // [1, 2]
     Object.entries(o3); // [["x",1],["y",2]]
     for (const [k, v] of Object.entries(o3)) { /* ... */ }

   Object.defineProperty(obj, key, descriptor) — precise control over a
   single property's behavior via a PROPERTY DESCRIPTOR:
     Object.defineProperty(obj, "id", {
       value: 42,
       writable: false,     // cannot be reassigned
       enumerable: false,   // won't show up in for...in / Object.keys / JSON.stringify
       configurable: false  // cannot be deleted or reconfigured
     });

   Object.getOwnPropertyDescriptor(obj, key) — inspect a descriptor.

────────────────────────────────────────────────────────────────────────────────
Q31: What are property descriptors, and what's the difference between
     data descriptors and accessor descriptors?
──────────────────────────────────────────────────────────────────────
A: Every property on an object has an underlying descriptor controlling its
   behavior, even ordinary properties created with `obj.x = 1` (which get
   default descriptors: writable/enumerable/configurable all `true`).

   DATA DESCRIPTOR — a property with a value:
     { value: 42, writable: true, enumerable: true, configurable: true }
   • value        — the actual property value.
   • writable     — can the value be changed by assignment?
   • enumerable   — does it show up in for...in, Object.keys, JSON.stringify?
   • configurable — can the descriptor itself be changed, or the property
                    deleted?

   ACCESSOR DESCRIPTOR — a property backed by get/set functions instead of a
   plain value:
     { get() { return this._x; }, set(v) { this._x = v; },
       enumerable: true, configurable: true }
   • Cannot have both a `value`/`writable` AND a `get`/`set` in the same
     descriptor — they're mutually exclusive.

     const obj = {};
     Object.defineProperty(obj, "readonly", {
       value: "fixed", writable: false, enumerable: true, configurable: false
     });
     obj.readonly = "changed"; // silently fails (sloppy) / throws (strict)
     console.log(obj.readonly); // "fixed"

   This is the mechanism underlying Object.freeze (sets writable: false,
   configurable: false on every property, and prevents extensions) and
   explains why some built-in properties (like Array.prototype.length or
   class methods) don't appear in for...in loops — they're non-enumerable.

SECTION 7: ARRAYS AND ITERATION
================================================================================

Q32: What are the most commonly used array methods? (map, filter, reduce,
     forEach, find, some/every, flat/flatMap, sort)
──────────────────────────────────────────────────────────────────────────
A: TRANSFORMATION / ITERATION (non-mutating unless noted):

     const nums = [1, 2, 3, 4, 5];

     nums.map(n => n * 2);              // [2,4,6,8,10] — new array, same length
     nums.filter(n => n % 2 === 0);     // [2,4] — new array, elements matching predicate
     nums.reduce((acc, n) => acc + n, 0); // 15 — folds array into a single value
     nums.forEach(n => console.log(n)); // undefined — just iterates, no return value
     nums.find(n => n > 3);             // 4 — first matching element (or undefined)
     nums.findIndex(n => n > 3);        // 3 — index of first match (or -1)
     nums.some(n => n > 4);             // true — at least one matches
     nums.every(n => n > 0);            // true — all match
     nums.includes(3);                  // true — value present (uses SameValueZero, handles NaN)

     [[1,2],[3,[4,5]]].flat();          // [1,2,3,[4,5]] — flattens 1 level deep by default
     [[1,2],[3,[4,5]]].flat(Infinity);  // [1,2,3,4,5] — fully flattens
     nums.flatMap(n => [n, n * 2]);      // [1,2,2,4,3,6,4,8,5,10] — map then flat(1), more efficient than chaining

     [3,1,2].sort();                     // [1,2,3] — MUTATES original! default sort is by STRING Unicode order
     [10,2,1].sort();                    // [1,10,2] — ⚠️ WRONG for numbers without a comparator!
     [10,2,1].sort((a,b) => a - b);      // [1,2,10] — ascending numeric comparator
     [10,2,1].sort((a,b) => b - a);      // [10,2,1] — descending

   MUTATING methods (change the original array): push, pop, shift, unshift,
   splice, sort, reverse, fill, copyWithin.

   NON-MUTATING methods (return a new array/value): map, filter, reduce,
   concat, slice, flat, flatMap, toSorted/toReversed/toSpliced (ES2023 —
   non-mutating counterparts of sort/reverse/splice).

     nums.slice(1, 3);   // [2,3] — non-mutating, extracts a range
     nums.splice(1, 2);   // removes 2 elements starting at index 1, MUTATES,
                          // returns the removed elements

────────────────────────────────────────────────────────────────────────────────
Q33: What is destructuring? Show object and array destructuring examples.
──────────────────────────────────────────────────────────────────────────
A: Destructuring (ES6) extracts values from arrays or properties from objects
   into distinct variables using a pattern that mirrors the source shape.

   ARRAY DESTRUCTURING (position-based):
     const [a, b, c] = [1, 2, 3];
     const [first, , third] = [1, 2, 3];        // skip elements with empty slots
     const [x, y = 10] = [5];                    // y = 10 (default value, x undefined→default)
     const [head, ...tail] = [1, 2, 3, 4];        // head=1, tail=[2,3,4]
     let m = 1, n = 2;
     [m, n] = [n, m];                              // swap without a temp variable → m=2, n=1

   OBJECT DESTRUCTURING (name-based):
     const { name, age } = { name: "Alice", age: 30 };
     const { name: userName } = { name: "Alice" };       // rename while destructuring
     const { role = "guest" } = {};                        // role = "guest" (default)
     const { a: { b } } = { a: { b: 42 } };                 // nested destructuring → b = 42

   Destructuring in function parameters (very common in React/config objects):
     function createUser({ name, age = 18, role = "user" } = {}) {
       return `${name}, ${age}, ${role}`;
     }
     createUser({ name: "Bob" }); // "Bob, 18, user"

   Combined with rest:
     const { id, ...otherProps } = { id: 1, name: "x", active: true };
     // otherProps = { name: "x", active: true }

────────────────────────────────────────────────────────────────────────────────
Q34: What is the spread operator and how does it differ from rest?
───────────────────────────────────────────────────────────────────
A: Both use the same `...` syntax but do OPPOSITE things depending on
   context:

   SPREAD — expands an iterable/object into individual elements:
     const arr1 = [1, 2, 3];
     const arr2 = [...arr1, 4, 5];        // [1,2,3,4,5] — new array (shallow copy + append)
     const combined = [...arr1, ...[6,7]]; // [1,2,3,6,7] — merge arrays

     const obj1 = { a: 1, b: 2 };
     const obj2 = { ...obj1, c: 3 };      // { a:1, b:2, c:3 } — shallow clone + merge
     const merged = { ...obj1, ...{ b: 99 } }; // { a:1, b:99 } — later spread wins on conflicts

     Math.max(...[1, 5, 3]);               // 5 — spread array into individual arguments
     function fn(a, b, c) {}
     fn(...[1, 2, 3]);                     // spreads array into 3 separate arguments

   REST — the inverse: COLLECTS multiple elements/arguments INTO a single
   array or object (used in destructuring patterns and function parameters):
     function sum(...nums) { return nums.reduce((a,b) => a+b, 0); } // rest parameter
     const [first, ...rest] = [1, 2, 3];   // rest = [2, 3] in destructuring
     const { a, ...others } = { a: 1, b: 2, c: 3 }; // others = { b:2, c:3 }

   Rule of thumb: spread appears on the RIGHT side / in a call (expanding);
   rest appears on the LEFT side / in a parameter list (collecting).
   Note both spread and Object.assign only do a SHALLOW copy — nested
   objects/arrays are still shared by reference.

────────────────────────────────────────────────────────────────────────────────
Q35: What is the difference between for...of and for...in?
────────────────────────────────────────────────────────────
A: for...in — iterates over the ENUMERABLE PROPERTY KEYS (string names) of an
   object, INCLUDING inherited enumerable properties from the prototype
   chain. Designed for plain objects, not recommended for arrays.

     const obj = { a: 1, b: 2 };
     for (const key in obj) {
       console.log(key); // "a", "b"
     }

     const arr = [10, 20, 30];
     for (const index in arr) {
       console.log(index); // "0", "1", "2" — STRINGS, and also picks up any
                            // enumerable custom properties added to the array
                            // or inherited via Array.prototype — unreliable for arrays
     }

   for...of — iterates over the VALUES produced by an ITERABLE (anything
   implementing the iterator protocol: arrays, strings, Maps, Sets,
   NodeLists, generators). This is the modern, preferred way to loop over
   arrays and other iterables.

     for (const value of [10, 20, 30]) {
       console.log(value); // 10, 20, 30 — actual values, correct type
     }
     for (const char of "hi") { console.log(char); } // "h", "i"
     for (const [k, v] of new Map([["a",1],["b",2]])) { /* [k,v] pairs */ }

     // for...of does NOT work directly on plain objects (not iterable):
     for (const x of { a: 1 }) {} // TypeError: object is not iterable
     // Use Object.entries(obj) / Object.keys(obj) / Object.values(obj) with
     // for...of instead.

   Summary: for...in → keys of an object (including inherited); for...of →
   values of an iterable. Prefer for...of (or array methods) for arrays.

────────────────────────────────────────────────────────────────────────────────
Q36: What are iterables and iterators? How does the iterator protocol work?
─────────────────────────────────────────────────────────────────────────────
A: The ITERATOR PROTOCOL defines a standard way to produce a sequence of
   values. An object is an ITERATOR if it has a `next()` method returning
   `{ value, done }`. An object is ITERABLE if it has a method at the key
   `Symbol.iterator` that returns an iterator.

   Manually implementing the iterator protocol:
     const range = {
       from: 1,
       to: 5,
       [Symbol.iterator]() {
         let current = this.from;
         const last = this.to;
         return {
           next() {
             return current <= last
               ? { value: current++, done: false }
               : { value: undefined, done: true };
           }
         };
       }
     };
     [...range];              // [1, 2, 3, 4, 5] — spread uses the iterator protocol
     for (const n of range) console.log(n); // for...of also uses it

   Built-in iterables: Array, String, Map, Set, TypedArray, arguments object,
   NodeList, and Generator objects.

   Anything relying on the iterator protocol automatically works with custom
   iterables: spread (...), destructuring, for...of, Array.from(),
   Promise.all(), yield* inside generators.

   Generators (function*) are the easiest way to create iterables — they
   automatically implement both next() and Symbol.iterator (see Section 13,
   Q56).

Q37: What is "callback hell" and how do Promises solve it?
───────────────────────────────────────────────────────────
A: Callback hell (the "pyramid of doom") happens when asynchronous operations
   are nested inside each other's callbacks, producing deeply indented,
   hard-to-read, hard-to-error-handle code:

     getUser(userId, function (user) {
       getOrders(user.id, function (orders) {
         getOrderDetails(orders[0].id, function (details) {
           getShipping(details.id, function (shipping) {
             console.log(shipping); // 4 levels deep, error handling duplicated
                                     // at every level
           }, handleError);
         }, handleError);
       }, handleError);
     }, handleError);

   Problems: hard to read (grows rightward), error handling repeated at
   every level, hard to compose/parallelize, "inversion of control" (you
   trust each callback-taking function to call your callback correctly,
   exactly once).

   Promises flatten this into a readable CHAIN, with a single centralized
   .catch() for errors:
     getUser(userId)
       .then(user => getOrders(user.id))
       .then(orders => getOrderDetails(orders[0].id))
       .then(details => getShipping(details.id))
       .then(shipping => console.log(shipping))
       .catch(handleError); // single error handler for the entire chain

   async/await (built on Promises) flattens it further into code that reads
   like synchronous code (see Q40).

────────────────────────────────────────────────────────────────────────────────
Q38: What are the states of a Promise? How does chaining work?
─────────────────────────────────────────────────────────────────
A: A Promise represents the eventual result of an asynchronous operation. It
   has exactly one of THREE states:
   • pending   — initial state, neither fulfilled nor rejected.
   • fulfilled — operation completed successfully, has a resulting value.
   • rejected  — operation failed, has a reason (usually an Error).

   Once settled (fulfilled OR rejected), a Promise's state and value are
   PERMANENT — cannot be changed again ("settled" is final).

     const p = new Promise((resolve, reject) => {
       setTimeout(() => {
         const success = true;
         success ? resolve("done!") : reject(new Error("failed"));
       }, 1000);
     });

   CHAINING — .then() ALWAYS returns a NEW Promise, which allows chaining.
   The value returned inside a .then() callback becomes the fulfillment
   value of the next Promise in the chain. If you return a Promise from
   inside .then(), the chain "flattens" and waits for it:

     fetchUser(1)
       .then(user => {
         console.log(user);
         return fetchOrders(user.id);   // returning a Promise — chain waits for it
       })
       .then(orders => {
         console.log(orders);
         return orders.length;           // returning a plain value
       })
       .then(count => console.log(count))
       .catch(err => console.error("Error in chain:", err)) // catches ANY
                                                              // rejection from
                                                              // above in the chain
       .finally(() => console.log("done, success or fail"));

   Throwing inside a .then() (or a rejected returned Promise) skips
   subsequent .then() handlers and jumps straight to the nearest .catch().

────────────────────────────────────────────────────────────────────────────────
Q39: What are Promise.all, Promise.allSettled, Promise.race, and Promise.any?
─────────────────────────────────────────────────────────────────────────────
A: These are static combinator methods for working with multiple Promises
   concurrently.

   Promise.all(promises) — waits for ALL to fulfill; resolves with an array
   of results in the SAME ORDER as input. If ANY promise rejects, Promise.all
   immediately rejects with that reason (fail-fast) — other still-pending
   promises keep running in the background but their results are ignored.
     const [users, orders] = await Promise.all([fetchUsers(), fetchOrders()]);
     // If fetchOrders() rejects, the whole Promise.all rejects immediately.

   Promise.allSettled(promises) — waits for ALL to SETTLE (fulfill or
   reject), NEVER rejects itself. Resolves with an array of result objects:
     [{ status: "fulfilled", value }, { status: "rejected", reason }, ...]
   Use when you want results from every promise regardless of individual
   failures.
     const results = await Promise.allSettled([fetchA(), fetchB()]);
     results.forEach(r => r.status === "fulfilled"
       ? console.log(r.value) : console.error(r.reason));

   Promise.race(promises) — settles as soon as the FIRST promise settles
   (whether fulfilled or rejected) — "whoever finishes first wins."
     Promise.race([fetchData(), timeout(5000)]) // classic timeout pattern

   Promise.any(promises) — settles as soon as the FIRST promise FULFILLS
   (ignores rejections unless ALL reject). If every promise rejects, it
   rejects with an AggregateError containing all individual errors.
     Promise.any([fetchFromMirror1(), fetchFromMirror2()]) // first successful wins

   Quick comparison table:
     Method            | Resolves when          | Rejects when
     ------------------|-------------------------|------------------------------
     Promise.all        | all fulfill             | any one rejects (fail-fast)
     Promise.allSettled | all settle (never rejects)| never
     Promise.race       | first settles (any way) | first settles as a rejection
     Promise.any         | first fulfills          | all reject (AggregateError)

────────────────────────────────────────────────────────────────────────────────
Q40: How does async/await work? How do you handle errors in async functions?
─────────────────────────────────────────────────────────────────────────────
A: async/await (ES2017) is syntactic sugar over Promises, allowing
   asynchronous code to be written and read like synchronous code.

   • `async function` always returns a Promise. If the function body returns
     a plain value, it's automatically wrapped in a resolved Promise; if it
     throws, the returned Promise is rejected with that error.
   • `await` pauses execution of the async function (without blocking the
     main thread/event loop) until the awaited Promise settles — unwrapping
     its resolved value, or throwing if it rejects.

     async function getUserData(id) {
       const user = await fetchUser(id);      // pauses here until resolved
       const orders = await fetchOrders(user.id);
       return { user, orders };                // wrapped in a resolved Promise
     }
     getUserData(1).then(data => console.log(data));

   ERROR HANDLING — use try/catch, since a rejected `await` throws:
     async function getUserData(id) {
       try {
         const user = await fetchUser(id);
         const orders = await fetchOrders(user.id);
         return { user, orders };
       } catch (err) {
         console.error("Failed to load user data:", err);
         throw err; // re-throw, or return a fallback value
       } finally {
         console.log("cleanup, always runs");
       }
     }

   Sequential vs parallel awaits (common performance pitfall):
     // SEQUENTIAL — slower, second call waits for first to finish unnecessarily
     const a = await fetchA(); // waits ~1s
     const b = await fetchB(); // then waits another ~1s → total ~2s

     // PARALLEL — starts both immediately, awaits together
     const [a2, b2] = await Promise.all([fetchA(), fetchB()]); // total ~1s

   An uncaught rejection inside an async function with no try/catch and no
   .catch() on the caller results in an "unhandled promise rejection."

────────────────────────────────────────────────────────────────────────────────
Q41: Explain the event loop. What is the difference between the microtask
     queue and the macrotask (task) queue?
──────────────────────────────────────────────────────────────────────────────
A: JavaScript is single-threaded — one call stack. The EVENT LOOP is the
   mechanism that allows non-blocking asynchronous behavior: it continuously
   checks whether the call stack is empty, and if so, pulls the next queued
   callback onto the stack to execute.

   There are two (relevant) queues, and they are NOT treated equally:

   MACROTASK QUEUE (a.k.a. "task queue" / "callback queue") — holds:
   • setTimeout / setInterval callbacks
   • I/O callbacks (Node.js)
   • UI rendering / requestAnimationFrame-ish browser events
   • setImmediate (Node.js only)

   MICROTASK QUEUE — holds:
   • Promise .then/.catch/.finally callbacks
   • queueMicrotask() callbacks
   • async/await continuations (code after an `await`)
   • MutationObserver callbacks

   THE ORDERING RULE (the key interview fact): after each single macrotask
   finishes running (including the very first synchronous script execution
   as the "first macrotask"), the event loop drains the ENTIRE microtask
   queue COMPLETELY — including any NEW microtasks scheduled by microtasks
   already running — before it proceeds to the next macrotask or to
   rendering. Microtasks always run before the next macrotask.

   Classic ordering example:
     console.log("1: sync start");

     setTimeout(() => console.log("2: setTimeout"), 0);

     Promise.resolve().then(() => console.log("3: promise 1"));

     Promise.resolve().then(() => {
       console.log("4: promise 2");
       Promise.resolve().then(() => console.log("5: nested promise"));
     });

     console.log("6: sync end");

     // Output order:
     // 1: sync start
     // 6: sync end
     // 3: promise 1
     // 4: promise 2
     // 5: nested promise      ← microtask queue fully drains, even a NEW
     //                           microtask scheduled mid-drain, before...
     // 2: setTimeout           ← ...the macrotask runs

   Why: synchronous code (1, 6) runs first as part of the initial script
   execution. Once the call stack is empty, ALL pending microtasks (3, 4,
   and the newly-queued 5) are drained before the engine even looks at the
   macrotask queue, so "2: setTimeout" — a macrotask — runs last, even
   though it was scheduled with a 0ms delay before some of the promise
   callbacks.

   setTimeout(fn, 0) does NOT mean "run immediately" — it means "run as soon
   as possible AFTER the current script and ALL pending microtasks finish,"
   with a practical minimum delay (browsers commonly clamp to ~4ms for
   nested timeouts).

────────────────────────────────────────────────────────────────────────────────
Q42: What is the difference between setTimeout and setInterval? What are
     common pitfalls?
──────────────────────────────────────────────────────────────────────────
A: setTimeout(fn, delay) — schedules `fn` to run ONCE, after AT LEAST `delay`
   milliseconds (not exactly — it's a minimum, actual delay depends on the
   call stack and queues being clear).
     const id = setTimeout(() => console.log("once"), 1000);
     clearTimeout(id); // cancels before it fires

   setInterval(fn, delay) — schedules `fn` to run REPEATEDLY every `delay`
   milliseconds until cancelled.
     const id2 = setInterval(() => console.log("tick"), 1000);
     clearInterval(id2); // must be explicitly cleared, or it runs forever

   Common pitfalls:
   • setInterval does NOT guarantee the delay between end of one execution
     and start of the next — if `fn` itself takes longer than `delay` to
     run, calls can queue up / overlap-adjust, causing "drift" or a burst of
     catch-up calls. A common safer pattern is recursive setTimeout:
       function poll() {
         doWork();
         setTimeout(poll, 1000); // schedules NEXT call only after current
                                  // work finishes — no overlap, no drift buildup
       }
       setTimeout(poll, 1000);
   • Forgetting to clearInterval causes a memory/CPU leak — the interval
     keeps firing even after the relevant component/page state is gone
     (classic React useEffect cleanup bug).
   • setTimeout(fn, 0) still queues `fn` as a macrotask — it runs after the
     current synchronous code AND after all pending microtasks, not
     literally "now" (see Q41).
   • `this` inside a regular function passed to setTimeout is NOT bound to
     any particular object by default (use an arrow function or .bind() if
     you need `this`).

Q43: What is the difference between CommonJS and ES Modules?
─────────────────────────────────────────────────────────────
A: COMMONJS (CJS) — Node.js's original module system.
     // math.js
     function add(a, b) { return a + b; }
     module.exports = { add };
     // or: exports.add = add;

     // app.js
     const { add } = require('./math');

   • Synchronous — require() loads and executes the module immediately,
     blocking until done. Fine for local file I/O in Node.
   • Dynamic — require() can be called conditionally/anywhere at runtime
     (e.g., inside an if block).
   • Exports a single `module.exports` object; values are COPIED at the time
     of `require()` for primitives (live bindings are not guaranteed the
     same way as ESM).
   • `this` at the top level refers to `module.exports`.

   ES MODULES (ESM) — the official standardized JS module system.
     // math.mjs
     export function add(a, b) { return a + b; }
     export default function multiply(a, b) { return a * b; }

     // app.mjs
     import multiply, { add } from './math.mjs';

   • Static structure — imports/exports are analyzed at parse time (before
     execution), enabling tree-shaking (bundlers can eliminate unused
     exports) and enabling `import` to be hoisted.
   • Asynchronous by design — supports top-level await and dynamic import().
   • Exports are LIVE BINDINGS — if the exporting module updates the
     variable later, importers see the updated value (not a stale copy).
   • Always in strict mode implicitly.
   • Enabled via `.mjs` extension, `"type": "module"` in package.json
     (Node), or `<script type="module">` in browsers.

   Interop: Node.js allows requiring CJS from ESM contexts with some
   restrictions; you cannot `require()` a genuine ESM module synchronously
   (must use dynamic `import()`).

────────────────────────────────────────────────────────────────────────────────
Q44: What is the difference between default exports and named exports?
────────────────────────────────────────────────────────────────────────
A: NAMED EXPORTS — a module can have MULTIPLE named exports. Importers must
   use the EXACT same name (or rename with `as`), wrapped in curly braces.

     // utils.js
     export const PI = 3.14159;
     export function square(x) { return x * x; }
     export class Circle { /* ... */ }

     // app.js
     import { PI, square, Circle } from './utils.js';
     import { square as sq } from './utils.js';       // rename on import
     import * as Utils from './utils.js';               // import everything as a namespace object
     Utils.square(4);

   DEFAULT EXPORT — a module can have AT MOST ONE default export. Importers
   can name it ANYTHING (no curly braces required) since there's no name to
   match against.

     // Logger.js
     export default class Logger { /* ... */ }

     // app.js
     import Logger from './Logger.js';      // any name works
     import MyLogger from './Logger.js';    // also valid, same import

   A module can mix both:
     export default function main() {}
     export const helper = () => {};
     // import main, { helper } from './module.js';

   Trade-offs:
   • Named exports: better for tooling (auto-import, refactor-rename works
     reliably), self-documenting, supports multiple exports naturally,
     easier to tree-shake individually.
   • Default exports: convenient for a module with one clear primary export
     (e.g., a single React component per file), but the free-naming can lead
     to inconsistent import names across a codebase and slightly worse
     tooling support historically. Many style guides now prefer named
     exports exclusively for consistency.

────────────────────────────────────────────────────────────────────────────────
Q45: What is dynamic import()? When would you use it?
────────────────────────────────────────────────────────
A: import() (as a function call, not the static `import` statement) loads a
   module ASYNCHRONOUSLY at runtime, returning a Promise that resolves to
   the module's namespace object. Unlike static imports, it can be called
   conditionally, in a loop, or in response to an event — anywhere
   expressions are allowed.

     button.addEventListener("click", async () => {
       const { default: Chart } = await import('./chart-library.js');
       new Chart(document.getElementById("canvas"));
     });

     // Conditional loading
     if (userWantsDarkMode) {
       const theme = await import('./dark-theme.js');
       theme.apply();
     }

   Use cases:
   • CODE SPLITTING — bundlers (webpack, Vite, Rollup) automatically split
     dynamically-imported modules into separate chunks, loaded on demand.
     Reduces initial bundle size / faster first load.
   • LAZY LOADING — load a heavy library (charting, PDF generation) only
     when the feature is actually used.
   • CONDITIONAL / ENVIRONMENT-SPECIFIC modules (e.g., load a polyfill only
     if a feature is missing).
   • Framework-level route-based code splitting (React.lazy() uses dynamic
     import() under the hood; Next.js/Vue Router do too).

     const LazyComponent = React.lazy(() => import('./MyComponent'));

Q46: What are the common ways to select and manipulate DOM elements?
───────────────────────────────────────────────────────────────────────
A: SELECTING elements:
     document.getElementById("app");                 // single element by id
     document.getElementsByClassName("item");        // live HTMLCollection
     document.getElementsByTagName("li");             // live HTMLCollection
     document.querySelector(".item.active");          // first match, any CSS selector
     document.querySelectorAll("ul > li");             // static NodeList, any CSS selector

   querySelector/querySelectorAll return a STATIC snapshot (NodeList);
   getElementsBy* return a LIVE collection that automatically updates as the
   DOM changes.

   MANIPULATING content/attributes:
     el.textContent = "Hello";        // sets text, safe from HTML injection
     el.innerHTML = "<b>Hi</b>";      // parses as HTML — XSS risk with user input!
     el.setAttribute("data-id", "5");
     el.getAttribute("data-id");
     el.classList.add("active");
     el.classList.remove("hidden");
     el.classList.toggle("open");
     el.style.color = "red";

   CREATING / INSERTING / REMOVING nodes:
     const div = document.createElement("div");
     div.textContent = "New";
     parent.appendChild(div);
     parent.insertBefore(div, referenceNode);
     parent.append(div, "text node too");   // modern, accepts multiple + strings
     el.remove();                            // removes itself from the DOM
     parent.removeChild(el);                 // older style

   Note: repeated DOM manipulation is expensive due to layout reflow —
   batch changes (e.g., via DocumentFragment) when inserting many nodes.

────────────────────────────────────────────────────────────────────────────────
Q47: What is event bubbling, capturing, and event delegation?
────────────────────────────────────────────────────────────────
A: When an event fires on an element, it doesn't just run handlers on that
   element — it propagates through the DOM tree in three phases:

   1. CAPTURING phase — the event travels DOWN from `window`/document
      through ancestors to the target element.
   2. TARGET phase — the event reaches the actual element that was
      interacted with.
   3. BUBBLING phase — the event travels back UP from the target through
      its ancestors to `window` (this is the DEFAULT phase handlers listen
      on).

     parent.addEventListener("click", () => console.log("parent"));      // bubble phase (default)
     child.addEventListener("click", () => console.log("child"));        // bubble phase (default)
     parent.addEventListener("click", () => console.log("parent capture"), true); // capture phase

     // Clicking `child` logs:
     // "parent capture"  (capturing: top-down)
     // "child"            (target)
     // "parent"           (bubbling: bottom-up)

   addEventListener(type, handler, useCaptureOrOptions) — third argument
   `true` (or `{ capture: true }`) registers the handler for the CAPTURING
   phase instead of the default bubbling phase.

   EVENT DELEGATION — instead of attaching a listener to every individual
   child element, attach ONE listener to a common ANCESTOR and use
   event.target to determine which child was actually interacted with. This
   leverages bubbling.

     // Instead of adding a click listener to every <li>:
     document.querySelector("ul").addEventListener("click", (e) => {
       if (e.target.matches("li")) {
         console.log("Clicked:", e.target.textContent);
       }
     });

   Benefits of delegation:
   • Fewer event listeners → better memory/performance, especially for long
     or dynamically-changing lists.
   • Automatically works for elements added to the DOM LATER (no need to
     re-attach listeners to new children).

────────────────────────────────────────────────────────────────────────────────
Q48: What is the difference between preventDefault() and stopPropagation()?
────────────────────────────────────────────────────────────────────────────
A: These solve two completely DIFFERENT problems and are often confused.

   event.preventDefault() — stops the BROWSER'S DEFAULT ACTION for that
   event from happening. Does NOT stop the event from propagating/bubbling
   to other handlers.
     form.addEventListener("submit", (e) => {
       e.preventDefault(); // stops the page from navigating/reloading on submit
       // ... handle form data via JS/fetch instead
     });
     link.addEventListener("click", (e) => {
       e.preventDefault(); // stops the browser from following the href
     });

   event.stopPropagation() — stops the event from CONTINUING to
   propagate/bubble (or capture) to ANCESTOR (or descendant) elements'
   listeners. Does NOT prevent the browser's default action.
     child.addEventListener("click", (e) => {
       e.stopPropagation(); // parent's click listener will NOT fire
     });

   event.stopImmediatePropagation() — like stopPropagation(), but ALSO
   prevents any OTHER listeners registered on the SAME element (for the same
   event) from running.

   These are independent — you can call one, both, or neither depending on
   what you need:
     el.addEventListener("click", (e) => {
       e.preventDefault();   // stop default browser action
       e.stopPropagation();  // AND stop bubbling to ancestors
     });

────────────────────────────────────────────────────────────────────────────────
Q49: What is the difference between localStorage, sessionStorage, and
     cookies?
──────────────────────────────────────────────────────────────────────
A: All three store data client-side (in the browser), but differ in
   capacity, lifetime, and whether they're sent to the server.

     Feature          | localStorage | sessionStorage | Cookies
     ------------------|--------------|-----------------|------------------------
     Capacity           | ~5-10 MB      | ~5-10 MB         | ~4 KB
     Lifetime            | Persists until explicitly cleared | Cleared when tab/window closes | Configurable expiry (or session)
     Sent to server?     | No — JS only  | No — JS only     | YES — sent with EVERY matching HTTP request (adds overhead)
     Scope                | Per origin (shared across tabs) | Per tab (not shared, even same origin) | Per domain + path, configurable
     Accessible via       | JS: localStorage.setItem/getItem | JS: sessionStorage.setItem/getItem | JS: document.cookie, or HttpOnly (server-only, JS-inaccessible)

   API examples:
     localStorage.setItem("theme", "dark");
     localStorage.getItem("theme");   // "dark"
     localStorage.removeItem("theme");
     localStorage.clear();

     sessionStorage.setItem("draft", JSON.stringify({ text: "hi" }));

     document.cookie = "sessionId=abc123; max-age=3600; path=/; Secure; SameSite=Strict";

   When to use which:
   • localStorage — user preferences, theme, non-sensitive data that should
     persist across sessions.
   • sessionStorage — temporary per-tab state (form drafts, wizard steps)
     that shouldn't survive closing the tab.
   • Cookies — data the SERVER needs on every request (auth session/JWT via
     HttpOnly cookies for XSS protection), or data needing fine-grained
     expiry and cross-request availability.
   • Sensitive tokens (auth) are best stored in HttpOnly, Secure, SameSite
     cookies — NOT localStorage, which is readable by any JS on the page and
     therefore vulnerable to XSS-based theft.

────────────────────────────────────────────────────────────────────────────────
Q50: How does the fetch API work? How do you handle errors correctly?
──────────────────────────────────────────────────────────────────────
A: fetch(url, options) returns a Promise that resolves to a Response object
   once headers are received (the body may still be streaming).

     async function getUser(id) {
       const response = await fetch(`/api/users/${id}`);
       if (!response.ok) {                     // ⚠️ critical gotcha, see below
         throw new Error(`HTTP error: ${response.status}`);
       }
       return await response.json();
     }

     // POST with a JSON body
     await fetch("/api/users", {
       method: "POST",
       headers: { "Content-Type": "application/json" },
       body: JSON.stringify({ name: "Alice" })
     });

   CRITICAL GOTCHA: fetch() does NOT reject its Promise on HTTP error status
   codes (404, 500, etc.) — it only rejects on NETWORK-level failures (DNS
   failure, no connectivity, CORS block). You MUST manually check
   `response.ok` (true for status 200-299) or `response.status` and throw/
   handle accordingly — otherwise a 404/500 page silently "succeeds" as far
   as the Promise chain is concerned.

     fetch("/api/might-404")
       .then(res => res.json())       // ⚠️ this "succeeds" even on a 404 —
       .then(data => console.log(data)); // parses the error page's JSON/HTML as if it were data

   Response body methods (each can only be consumed ONCE):
     response.json()   // parse as JSON
     response.text()   // raw text
     response.blob()   // binary data (files/images)
     response.formData()

   Aborting a fetch (timeout pattern):
     const controller = new AbortController();
     setTimeout(() => controller.abort(), 5000);
     fetch(url, { signal: controller.signal })
       .catch(err => { if (err.name === "AbortError") console.log("timed out"); });

────────────────────────────────────────────────────────────────────────────────
Q51: What is the difference between JSON.parse() and JSON.stringify()?
───────────────────────────────────────────────────────────────────────
A: JSON.stringify(value, replacer?, space?) — converts a JS value into a
   JSON-formatted STRING.
     JSON.stringify({ a: 1, b: [1,2,3] });         // '{"a":1,"b":[1,2,3]}'
     JSON.stringify({ a: 1 }, null, 2);              // pretty-printed with 2-space indent
     JSON.stringify({ a: undefined, b: function(){}, c: Symbol() }); // '{}' — these are SKIPPED (not valid JSON)
     JSON.stringify([undefined, function(){}]);       // '[null,null]' — in arrays, become null instead of being skipped
     JSON.stringify({ a: 1, b: 2 }, ["a"]);            // '{"a":1}' — replacer array whitelist
     JSON.stringify(new Date());                       // ISO string, via Date.prototype.toJSON()
     JSON.stringify({ a: undefined ? 1 : NaN });         // NaN and Infinity both serialize to "null"

   JSON.parse(text, reviver?) — parses a JSON string back into a JS value.
     JSON.parse('{"a":1,"b":[1,2,3]}');   // { a: 1, b: [1,2,3] }
     JSON.parse('not json');               // throws SyntaxError

     // Reviver function — transform values during parsing
     JSON.parse('{"date":"2024-01-01"}', (key, value) =>
       key === "date" ? new Date(value) : value);

   Common practical use — deep cloning a plain, JSON-safe object (has real
   limitations):
     const clone = JSON.parse(JSON.stringify(original));
     // ⚠️ Loses: functions, undefined values, Symbols, Dates (become strings),
     // Maps/Sets (become {}), and throws on circular references.
     // Prefer structuredClone(original) (modern browsers/Node 17+) for a
     // proper deep clone that handles more types and circular refs correctly.

Q52: How does try/catch/finally work? What can you throw?
──────────────────────────────────────────────────────────
A: try { } — code that might throw an error.
   catch (err) { } — runs if an error is thrown inside try; `err` holds the
   thrown value. The parameter is optional in modern JS: `catch { }`.
   finally { } — ALWAYS runs, whether an error was thrown or not, and even
   if try/catch contains a `return` statement (finally runs before the
   function actually returns).

     function process() {
       try {
         riskyOperation();
         return "success";
       } catch (err) {
         console.error(err.message);
         return "failed";
       } finally {
         console.log("cleanup runs no matter what"); // always executes
       }
     }

   You can `throw` ANY value in JavaScript — not just Error objects:
     throw "a string error";     // works, but bad practice
     throw 42;                    // works, but bad practice
     throw { code: 500 };         // works, but bad practice
     throw new Error("proper");   // ✅ best practice — gives .message, .stack, .name

   Always throw Error objects (or subclasses) — they capture a STACK TRACE
   at the point of creation (`err.stack`), which is invaluable for
   debugging. Throwing plain strings/objects loses this.

   finally + return interaction gotcha: if `finally` itself contains a
   `return`, it OVERRIDES any return/throw from the try/catch block:
     function weird() {
       try { return 1; }
       finally { return 2; } // 2 wins — overrides the try's return 1
     }
     weird(); // 2

────────────────────────────────────────────────────────────────────────────────
Q53: How do you create and use custom Error types?
─────────────────────────────────────────────────────
A: Extend the built-in Error class to create domain-specific error types with
   extra metadata, enabling targeted handling with instanceof checks.

     class ValidationError extends Error {
       constructor(message, field) {
         super(message);              // sets this.message
         this.name = "ValidationError"; // overrides default "Error" name
         this.field = field;           // custom metadata
         // Ensures correct prototype chain in some transpiled/older environments:
         Object.setPrototypeOf(this, ValidationError.prototype);
         if (Error.captureStackTrace) {
           Error.captureStackTrace(this, ValidationError); // V8: excludes
                                                             // constructor from stack trace
         }
       }
     }

     class NotFoundError extends Error {
       constructor(resource) {
         super(`${resource} not found`);
         this.name = "NotFoundError";
       }
     }

     function validateAge(age) {
       if (age < 0) throw new ValidationError("Age cannot be negative", "age");
     }

     try {
       validateAge(-5);
     } catch (err) {
       if (err instanceof ValidationError) {
         console.log(`Validation failed on field: ${err.field}`);
       } else if (err instanceof NotFoundError) {
         console.log("Not found:", err.message);
       } else {
         throw err; // re-throw unknown errors — don't swallow silently
       }
     }

   Built-in Error subtypes: TypeError, RangeError, ReferenceError,
   SyntaxError, URIError, EvalError, AggregateError (ES2021, wraps multiple
   errors — used by Promise.any).

   ES2022 error cause chaining — preserves the original error when
   re-wrapping:
     try {
       JSON.parse(badInput);
     } catch (err) {
       throw new Error("Failed to parse config", { cause: err });
     }
     // err.cause gives access to the original underlying error

────────────────────────────────────────────────────────────────────────────────
Q54: How do you correctly handle errors in asynchronous code (callbacks,
     Promises, async/await)?
─────────────────────────────────────────────────────────────────────────────
A: Each async pattern has its own error-handling convention:

   CALLBACKS — "error-first callback" convention (Node.js style): the first
   argument is reserved for an error (or null if none).
     fs.readFile("file.txt", (err, data) => {
       if (err) return console.error("Failed:", err);
       console.log(data);
     });
   A try/catch WRAPPED AROUND an async callback-based call does NOT catch
   errors that happen inside the callback later — the try block has already
   exited by the time the callback runs:
     try {
       setTimeout(() => { throw new Error("boom"); }, 100); // NOT caught below
     } catch (err) {
       console.log("never runs"); // the throw happens asynchronously, outside this try
     }

   PROMISES — use .catch() at the end of a chain; it catches rejections from
   ANY earlier .then() in the chain (including synchronous throws inside a
   .then() callback, which are automatically converted into a rejection).
     doSomething()
       .then(result => mightThrow(result))
       .catch(err => console.error("Caught:", err)); // catches errors from
                                                        // doSomething() OR mightThrow()

   ASYNC/AWAIT — use try/catch, since `await` on a rejected Promise THROWS
   synchronously within the async function body (see Q40).

   UNHANDLED REJECTIONS — a Promise that rejects with no .catch() anywhere
   in its chain triggers a global "unhandledrejection" event (browser) or
   process warning/crash (Node, depending on version/config). Always attach
   error handling to every Promise chain, or wrap top-level async calls:
     main().catch(err => console.error("Fatal:", err));

────────────────────────────────────────────────────────────────────────────────
Q55: What is the Error object's structure? What properties does it have?
─────────────────────────────────────────────────────────────────────────
A: The built-in Error object (and its subclasses) has these standard
   properties:
   • message — human-readable description of the error.
   • name    — the error type name ("Error", "TypeError", etc.), used by
     err.toString() as `${name}: ${message}`.
   • stack   — a non-standard but universally-implemented string containing
     the call stack at the point the error was created (invaluable for
     debugging; format varies by engine).
   • cause   — (ES2022) optional, holds an underlying error when wrapping/
     re-throwing, set via `new Error(msg, { cause: original })`.

     try {
       null.foo;
     } catch (err) {
       console.log(err.name);    // "TypeError"
       console.log(err.message); // "Cannot read properties of null (reading 'foo')"
       console.log(err.stack);   // full stack trace string
     }

   Note that `err.stack` is captured at OBJECT CREATION time (`new Error()`),
   not at throw time — so `const e = new Error(); setTimeout(() => throw e)`
   captures the stack from where `new Error()` was called, not from the
   setTimeout callback's context.

SECTION 12: MEMORY AND PERFORMANCE
================================================================================

Q56: How does garbage collection work in JavaScript?
───────────────────────────────────────────────────────
A: JavaScript uses AUTOMATIC memory management — you don't manually allocate
   or free memory. The garbage collector (GC) periodically reclaims memory
   for objects that are no longer REACHABLE.

   REACHABILITY is the core concept: an object is reachable (and therefore
   kept alive) if it can be accessed, directly or through a chain of
   references, starting from a set of "roots" — global variables, the
   current call stack's local variables/closures, and so on. Objects that
   are NOT reachable from any root are eligible for garbage collection.

   The dominant algorithm modern engines (V8, SpiderMonkey) use is
   MARK-AND-SWEEP:
   1. MARK phase — starting from the roots, the GC walks the entire graph of
      references, marking every object it can reach as "alive."
   2. SWEEP phase — any object NOT marked is considered garbage and its
      memory is reclaimed.

   This correctly handles CIRCULAR REFERENCES (two objects referencing each
   other but unreachable from any root) — unlike the older, simpler
   "reference counting" algorithm, which leaks memory on cycles because each
   object's count never reaches zero.

     function createCycle() {
       const a = {};
       const b = {};
       a.ref = b;
       b.ref = a;
       return null; // both a and b become unreachable from any root once
                      // createCycle() returns — mark-and-sweep correctly
                      // collects them despite the cycle
     }

   V8 specifically uses a GENERATIONAL GC — splits the heap into a small
   "young generation" (new objects, collected frequently/cheaply via a fast
   "Scavenger" algorithm) and a larger "old generation" (long-lived objects
   that survived several young-gen collections, collected less often via a
   more thorough mark-sweep-compact pass) — because most objects die young
   (the "generational hypothesis").

────────────────────────────────────────────────────────────────────────────────
Q57: What are common causes of memory leaks in JavaScript?
───────────────────────────────────────────────────────────
A: Since GC only frees UNREACHABLE memory, leaks happen when references are
   unintentionally kept alive longer than needed.

   1. ACCIDENTAL GLOBAL VARIABLES — forgetting `var`/`let`/`const` in
      non-strict mode creates an implicit global that never gets cleaned up:
        function leaky() { accidentalGlobal = "I'm global now"; } // no declaration keyword

   2. FORGOTTEN TIMERS/INTERVALS — an interval that references
      DOM elements/closures and is never cleared keeps them alive forever:
        function startPolling() {
          const bigData = fetchHugeDataset();
          setInterval(() => useData(bigData), 1000); // never cleared → bigData
                                                        // and the interval live forever
        }
      Fix: always clearInterval/clearTimeout when done (e.g., in a React
      useEffect cleanup function).

   3. DETACHED DOM NODES — keeping a JS reference to a DOM element after
      removing it from the document; the element (and everything it
      references) can't be collected because your variable still points to
      it:
        let cachedRow = document.getElementById("row-5");
        cachedRow.remove(); // removed from DOM, but NOT garbage collected —
                              // `cachedRow` variable still references it

   4. UNREMOVED EVENT LISTENERS — attaching listeners to long-lived objects
      (window, document) that reference short-lived objects in their
      closure, without ever calling removeEventListener.
        function attach() {
          const largeObject = { /* ... */ };
          window.addEventListener("resize", () => console.log(largeObject));
          // largeObject is kept alive for the lifetime of the page unless
          // the listener is removed
        }

   5. CLOSURES holding references longer than intended — a closure
      capturing a large object when it only needed one small property from
      it.

   6. CACHES/MAPS growing unbounded — a plain Map/object used as a cache
      that keys on objects and never evicts entries, preventing those
      keyed objects from ever being collected (solved by WeakMap, see Q59).

   Detecting leaks: browser DevTools → Memory tab → take heap snapshots over
   time and compare; look for growing counts of "detached" DOM nodes or
   ever-increasing retained size.

────────────────────────────────────────────────────────────────────────────────
Q58: What is the difference between debounce and throttle? Implement both.
───────────────────────────────────────────────────────────────────────────
A: Both limit how often a function runs in response to rapidly-firing events
   (scroll, resize, keystroke, mousemove), but with different strategies.

   DEBOUNCE — delays execution until AFTER a burst of calls has STOPPED for
   a specified quiet period. Every new call RESETS the timer. Good for:
   search-as-you-type (wait until the user pauses typing), window resize
   (wait until resizing stops), form validation on input.

     function debounce(fn, delay) {
       let timeoutId;
       return function (...args) {
         clearTimeout(timeoutId);                  // cancel the previous pending call
         timeoutId = setTimeout(() => fn.apply(this, args), delay);
       };
     }

     const debouncedSearch = debounce((query) => fetchResults(query), 300);
     input.addEventListener("input", (e) => debouncedSearch(e.target.value));
     // Rapid keystrokes: only fires ONCE, 300ms after the LAST keystroke.

   THROTTLE — ensures the function runs AT MOST once per specified time
   interval, no matter how many times it's called, executing at a regular
   cadence during continuous activity. Good for: scroll position tracking,
   mousemove-based drag handling, rate-limiting button clicks/API calls.

     function throttle(fn, interval) {
       let lastCallTime = 0;
       return function (...args) {
         const now = Date.now();
         if (now - lastCallTime >= interval) {
           lastCallTime = now;
           fn.apply(this, args);
         }
       };
     }

     const throttledScroll = throttle(() => console.log(window.scrollY), 200);
     window.addEventListener("scroll", throttledScroll);
     // Fires at most once every 200ms while scrolling continuously.

   Quick comparison:
     Debounce → "wait for a pause, then act once" (search boxes, autosave).
     Throttle → "act at a steady maximum rate" (scroll/resize tracking,
                infinite-scroll triggers, drag events).

────────────────────────────────────────────────────────────────────────────────
Q59: What are WeakMap and WeakSet? Why do they help with garbage collection?
─────────────────────────────────────────────────────────────────────────────
A: WeakMap and WeakSet are collections whose KEYS (WeakMap) or VALUES
   (WeakSet) must be OBJECTS, and are held with WEAK REFERENCES — meaning
   they do NOT prevent those objects from being garbage collected.

   Regular Map/Set hold STRONG references — an object used as a Map key
   stays alive for as long as the Map exists, even if nothing else in the
   program references it anymore. This can cause memory leaks in
   long-lived caches.

     const cache = new Map();
     function process(obj) {
       cache.set(obj, expensiveComputation(obj));
     }
     // If `obj` is later discarded everywhere else in the app, it is STILL
     // kept alive by `cache` forever — leak, unless manually deleted.

   WeakMap fixes this:
     const wCache = new WeakMap();
     function process(obj) {
       wCache.set(obj, expensiveComputation(obj));
     }
     // If `obj` becomes unreachable elsewhere, the GC can collect it AND
     // automatically remove its entry from wCache — no leak, no manual cleanup.

   Key restrictions (trade-offs for this GC-friendliness):
   • Keys/values MUST be objects (or non-registered Symbols) — cannot use
     primitives (strings, numbers) as WeakMap keys.
   • NOT iterable — no .keys()/.values()/.entries()/.forEach()/.size,
     because the contents can change at any time as the GC runs (iterating
     over something that can shrink unpredictably would be unreliable).
   • Only has: get, set, has, delete (WeakMap) / add, has, delete (WeakSet).

   Common use cases:
   • Associating private/extra metadata with objects you don't own without
     modifying them or leaking memory (e.g., DOM element metadata caches).
   • Implementing true private data for objects before/alongside class
     `#privateFields`.
   • Caching computed results keyed by object identity (memoization) that
     should self-clean when the object goes away.

Q61: What is optional chaining (?.) and nullish coalescing (??)?
─────────────────────────────────────────────────────────────────
A: OPTIONAL CHAINING (?.) (ES2020) — safely accesses a nested property,
   short-circuiting to `undefined` instead of throwing if any link in the
   chain is `null` or `undefined`.

     const user = { profile: { address: null } };
     user.profile.address.city;      // TypeError: Cannot read properties of null
     user.profile.address?.city;      // undefined — safely short-circuits
     user.profile?.address?.city;     // undefined
     user.nonExistent?.foo?.bar;      // undefined — no error even though
                                        // `nonExistent` doesn't exist at all

     // Works with method calls too — checks the method exists before calling:
     user.greet?.();                  // undefined if greet doesn't exist,
                                        // instead of "greet is not a function"

     // Works with array/bracket access:
     arr?.[0];
     obj?.["dynamic-key"];

   NULLISH COALESCING (??) (ES2020) — returns the RIGHT-hand operand ONLY
   when the left-hand operand is `null` or `undefined` — NOT for other
   falsy values (0, "", false, NaN). This is the key difference from `||`.

     const count = 0;
     count || 10;    // 10 — ⚠️ WRONG if 0 is a valid value! || treats 0 as falsy
     count ?? 10;    // 0  — ✅ correct — 0 is neither null nor undefined

     const name = "";
     name || "Guest"; // "Guest" — ⚠️ treats empty string as "missing"
     name ?? "Guest"; // ""      — ✅ empty string is a valid, intentional value

   Combined, these two operators together dramatically reduce defensive code
   for accessing possibly-missing nested config/API data:
     const theme = response.data?.settings?.theme ?? "light";

   Note: `??` cannot be directly mixed with `||` or `&&` in the same
   expression without parentheses — `a || b ?? c` is a SyntaxError, must
   write `(a || b) ?? c`.

────────────────────────────────────────────────────────────────────────────────
Q62: What are generators? What does the yield keyword do?
──────────────────────────────────────────────────────────
A: A GENERATOR is a special function (declared with `function*`) that can
   PAUSE its execution mid-way and later RESUME from exactly where it left
   off — unlike a normal function, which always runs start-to-finish in one
   go. Calling a generator function does NOT run its body immediately —
   instead it returns a GENERATOR OBJECT, which is both an iterator and an
   iterable.

     function* countUpTo(max) {
       let i = 1;
       while (i <= max) {
         yield i;   // pauses here, returns `i`, waits to be resumed
         i++;
       }
       return "done";
     }

     const gen = countUpTo(3);
     gen.next(); // { value: 1, done: false }
     gen.next(); // { value: 2, done: false }
     gen.next(); // { value: 3, done: false }
     gen.next(); // { value: "done", done: true } — the return value comes through here

     // Generators are iterable — usable directly in for...of and spread:
     for (const n of countUpTo(3)) console.log(n); // 1, 2, 3 (does not include the return value)
     [...countUpTo(3)]; // [1, 2, 3]

   `yield` can also RECEIVE a value passed into the NEXT .next(call) —
   enabling two-way communication:
     function* echo() {
       const received = yield "first";
       console.log("received:", received);
     }
     const it = echo();
     it.next();          // { value: "first", done: false }
     it.next("hello");   // logs "received: hello"

   `yield*` DELEGATES to another iterable/generator, forwarding all its
   values:
     function* inner() { yield 1; yield 2; }
     function* outer() { yield* inner(); yield 3; }
     [...outer()]; // [1, 2, 3]

   Use cases: implementing custom iterables, lazy-evaluated infinite
   sequences, simplifying complex async control flow (historically, before
   async/await existed, generators + a runner library were used to write
   async code synchronously — redux-saga still uses this pattern today).

────────────────────────────────────────────────────────────────────────────────
Q63: What are Proxy and Reflect? What can you do with them?
────────────────────────────────────────────────────────────
A: PROXY (ES6) wraps a target object and lets you intercept and customize
   fundamental operations on it (property access, assignment, deletion,
   function calls, etc.) via "traps."

     const target = { name: "Alice", age: 30 };
     const handler = {
       get(obj, prop) {
         console.log(`Getting ${prop}`);
         return prop in obj ? obj[prop] : `${prop} does not exist`;
       },
       set(obj, prop, value) {
         if (prop === "age" && typeof value !== "number") {
           throw new TypeError("age must be a number");
         }
         obj[prop] = value;
         return true; // must return true to indicate success
       }
     };
     const proxy = new Proxy(target, handler);
     proxy.name;          // logs "Getting name", returns "Alice"
     proxy.unknown;        // logs "Getting unknown", returns "unknown does not exist"
     proxy.age = "old";    // throws TypeError

   Common traps: get, set, has (for `in` operator), deleteProperty,
   ownKeys, apply (function calls), construct (`new` calls),
   defineProperty.

   Use cases: validation layers, reactive frameworks (Vue 3's reactivity
   system is built on Proxy instead of Object.defineProperty used in Vue 2),
   logging/auditing property access, negative array indexing, default
   values for missing properties, API mocking.

   REFLECT is a built-in object providing methods that mirror the internal
   operations Proxy traps intercept — essentially the "default behavior"
   counterparts (Reflect.get, Reflect.set, Reflect.has, Reflect.deleteProperty,
   etc.). It's commonly used INSIDE Proxy traps to invoke the default
   behavior after custom logic runs, cleanly forwarding the operation:

     const handler2 = {
       get(obj, prop, receiver) {
         console.log(`Accessed ${String(prop)}`);
         return Reflect.get(obj, prop, receiver); // performs the default get,
                                                     // correctly respecting `receiver`/prototype chain
       }
     };

   Reflect methods return sensible values (e.g., booleans for success/
   failure) instead of throwing, making them more predictable to use
   programmatically than the equivalent Object.* methods or operators.

────────────────────────────────────────────────────────────────────────────────
Q64: What is BigInt? When would you use it?
───────────────────────────────────────────
A: BigInt (ES2020) is a primitive type for representing integers of
   ARBITRARY precision — beyond the safe range of the `number` type, which
   is limited to Number.MAX_SAFE_INTEGER (2^53 - 1 = 9007199254740991) due
   to using IEEE-754 double-precision floating point for all numbers.

     Number.MAX_SAFE_INTEGER;             // 9007199254740991
     Number.MAX_SAFE_INTEGER + 1;          // 9007199254740992 — still "correct" by luck
     Number.MAX_SAFE_INTEGER + 2;          // 9007199254740992 — ⚠️ WRONG, precision lost!

     const big = 9007199254740993n;        // trailing `n` creates a BigInt literal
     const big2 = BigInt(9007199254740993); // or via the BigInt() function/constructor
     big + 1n;                               // 9007199254740994n — exact, no precision loss

   Rules and gotchas:
   • Cannot mix BigInt and Number in arithmetic directly — throws TypeError:
       1n + 1;      // TypeError: Cannot mix BigInt and other types
       1n + BigInt(1); // 2n — must explicitly convert
   • typeof 10n === "bigint" (a distinct primitive type, not "number").
   • BigInt == Number comparisons DO work (loose equality coerces):
       10n == 10;   // true
       10n === 10;  // false — different types, strict equality fails
   • No decimals — BigInt is integers only; division truncates: 7n / 2n === 3n.
   • Cannot be used with Math object methods.
   • Not serializable by JSON.stringify() by default — throws TypeError.

   Use cases: cryptography, high-precision timestamps/IDs (e.g., Twitter/X
   snowflake IDs exceed 2^53), large financial calculations, working with
   64-bit integers from other systems/databases.

Q65: Implement the Module Pattern and the Singleton Pattern in JavaScript.
──────────────────────────────────────────────────────────────────────────
A: MODULE PATTERN — uses an IIFE + closures to create private state and
   expose only a public API, achieving encapsulation before ES6 modules
   existed (still useful conceptually and in non-module scripts):

     const CounterModule = (function () {
       let count = 0;                       // private — inaccessible from outside
       function validate(n) { return typeof n === "number"; }  // private helper

       return {                              // public API
         increment() { count++; return count; },
         decrement() { count--; return count; },
         getCount() { return count; }
       };
     })();

     CounterModule.increment();  // 1
     CounterModule.count;         // undefined — truly private, not just convention

   Modern equivalent: an ES module file naturally provides this via
   module-level `let`/`const` (not exported) plus explicit `export`s.

   SINGLETON PATTERN — ensures a class/object has only ONE instance,
   globally accessible.

     class Database {
       static #instance;          // private static field (modern syntax)
       constructor() {
         if (Database.#instance) {
           return Database.#instance;   // return existing instance instead of a new one
         }
         this.connection = "connected";
         Database.#instance = this;
       }
       static getInstance() {
         if (!Database.#instance) {
           Database.#instance = new Database();
         }
         return Database.#instance;
       }
     }
     const db1 = new Database();
     const db2 = new Database();
     db1 === db2; // true — same instance

     // Simpler singleton via a module: since ES modules are cached after
     // first import, a module's top-level exported object IS a singleton
     // by nature — no special pattern code needed:
     // config.js: export default { apiUrl: "..." };  ← same object everywhere it's imported

────────────────────────────────────────────────────────────────────────────────
Q66: Implement the Observer Pattern (a simple Event Emitter) in JavaScript.
─────────────────────────────────────────────────────────────────────────────
A: The Observer pattern lets objects ("subscribers"/"listeners") register
   interest in events and get notified when those events occur, without the
   publisher needing to know who's listening — this is exactly what
   addEventListener/EventTarget and Node's EventEmitter provide natively.

     class EventEmitter {
       #listeners = new Map();  // eventName -> Set of callback functions

       on(event, callback) {
         if (!this.#listeners.has(event)) {
           this.#listeners.set(event, new Set());
         }
         this.#listeners.get(event).add(callback);
         return this; // enable chaining: emitter.on("a", fn1).on("b", fn2)
       }

       off(event, callback) {
         this.#listeners.get(event)?.delete(callback);
         return this;
       }

       once(event, callback) {
         const wrapper = (...args) => {
           callback(...args);
           this.off(event, wrapper);
         };
         return this.on(event, wrapper);
       }

       emit(event, ...args) {
         this.#listeners.get(event)?.forEach(callback => callback(...args));
         return this;
       }
     }

     const emitter = new EventEmitter();
     function onUserCreated(user) { console.log(`Welcome, ${user.name}!`); }

     emitter.on("userCreated", onUserCreated);
     emitter.emit("userCreated", { name: "Alice" }); // "Welcome, Alice!"
     emitter.off("userCreated", onUserCreated);
     emitter.emit("userCreated", { name: "Bob" });    // nothing happens — unsubscribed

   This is the foundation of: DOM events, Node.js's EventEmitter/streams,
   Redux store subscriptions, and pub/sub architectures in general.

────────────────────────────────────────────────────────────────────────────────
Q67: Implement your own version of Function.prototype.bind (a polyfill).
─────────────────────────────────────────────────────────────────────────
A: Understanding bind's polyfill tests grasp of `this`, closures, apply(),
   and the `new` interaction edge case.

     Function.prototype.myBind = function (thisArg, ...boundArgs) {
       const originalFn = this; // the function myBind was called on

       function boundFn(...callArgs) {
         // If boundFn is called with `new`, `this` should be the new
         // instance being constructed, NOT the bound thisArg — `new`
         // always takes precedence over bind's thisArg.
         const isNewCall = this instanceof boundFn;
         return originalFn.apply(
           isNewCall ? this : thisArg,
           [...boundArgs, ...callArgs]
         );
       }

       // Preserve the prototype chain so `instanceof` works correctly
       // when boundFn is used as a constructor.
       boundFn.prototype = Object.create(originalFn.prototype || {});
       return boundFn;
     };

     function greet(greeting, punctuation) {
       return `${greeting}, ${this.name}${punctuation}`;
     }
     const bound = greet.myBind({ name: "Alice" }, "Hello");
     bound("!"); // "Hello, Alice!"

────────────────────────────────────────────────────────────────────────────────
Q68: Implement your own version of Array.prototype.map (a polyfill).
─────────────────────────────────────────────────────────────────────
A: Array.prototype.myMap = function (callback, thisArg) {
     if (typeof callback !== "function") {
       throw new TypeError(`${callback} is not a function`);
     }
     const result = [];
     for (let i = 0; i < this.length; i++) {
       if (i in this) {                       // skip holes in sparse arrays,
                                                // matching native map's behavior
         result[i] = callback.call(thisArg, this[i], i, this);
       }
     }
     return result;
   };

   const doubled = [1, 2, 3].myMap(x => x * 2); // [2, 4, 6]

   Key details a strong answer should mention: `this` inside myMap refers to
   the array it's called on; the callback receives (element, index, array);
   an optional thisArg lets the caller control `this` inside the callback;
   native implementations skip "holes" in sparse arrays ([1,,3]), which a
   naive `this.map ? ... : ...` for-loop must replicate with the `i in this`
   check to be fully spec-accurate.

────────────────────────────────────────────────────────────────────────────────
Q69: Predict the output — closures inside loops (var vs let).
──────────────────────────────────────────────────────────────
A: This is one of the most common "predict the output" interview questions.

     for (var i = 0; i < 3; i++) {
       setTimeout(() => console.log("var:", i), 10);
     }
     // Output: "var: 3", "var: 3", "var: 3"
     // Why: `var` is function/global-scoped — there's only ONE `i` shared by
     // all three closures. By the time the setTimeout callbacks actually
     // run (after the synchronous loop has fully finished), i === 3.

     for (let j = 0; j < 3; j++) {
       setTimeout(() => console.log("let:", j), 10);
     }
     // Output: "let: 0", "let: 1", "let: 2"
     // Why: `let` creates a BRAND NEW binding of `j` for EACH loop
     // iteration (this is defined in the spec specifically for `for`
     // loops with let/const) — each closure captures its own independent
     // snapshot.

   Pre-ES6 fix for the `var` case — force a new scope per iteration using an
   IIFE:
     for (var k = 0; k < 3; k++) {
       (function (capturedK) {
         setTimeout(() => console.log("iife:", capturedK), 10);
       })(k);
     }
     // Output: "iife: 0", "iife: 1", "iife: 2" — capturedK is a fresh
     // parameter per IIFE call, achieving the same effect `let` gives natively.

────────────────────────────────────────────────────────────────────────────────
Q70: Predict the output — tricky async ordering with setTimeout, Promises,
     and synchronous code together.
──────────────────────────────────────────────────────────────────────────────
A: console.log("A");

     setTimeout(() => console.log("B"), 0);

     Promise.resolve()
       .then(() => console.log("C"))
       .then(() => console.log("D"));

     console.log("E");

     async function asyncFn() {
       console.log("F");
       await null;
       console.log("G");
     }
     asyncFn();

     console.log("H");

   Output: A, E, F, H, C, G, D, B

   Step-by-step reasoning:
   1. "A" — synchronous, logs immediately.
   2. setTimeout callback is registered as a MACROTASK — deferred to the end.
   3. Promise.resolve().then(...) registers a MICROTASK for "C" (the .then
      chain's SECOND .then for "D" isn't queued yet — it only gets queued
      once the FIRST .then's callback actually runs and returns).
   4. "E" — synchronous, logs immediately.
   5. asyncFn() is called: runs synchronously up to the first `await` —
      logs "F" synchronously, THEN `await null` suspends the function and
      queues the remainder ("G") as a MICROTASK.
   6. "H" — synchronous, logs immediately (asyncFn() is non-blocking).
   7. Synchronous script now fully done. Call stack empty → drain microtask
      queue IN ORDER: "C" runs first (queued in step 3), which SCHEDULES a
      new microtask for "D"; then "G" runs (queued in step 5); then "D"
      runs (queued during step 7's "C" execution) — all still within the
      SAME microtask-draining pass, before any macrotask.
   8. Microtask queue now empty → event loop proceeds to the macrotask
      queue → "B" (setTimeout) finally runs, LAST.

   The universal rule to remember: ALL synchronous code first, THEN ALL
   microtasks (Promises/async continuations) fully drain (including new
   ones queued during the drain), THEN the next macrotask (setTimeout/
   setInterval).

────────────────────────────────────────────────────────────────────────────────
Q71: What is functional composition and immutability? How do they relate to
     functional programming in JS?
──────────────────────────────────────────────────────────────────────────────
A: FUNCTIONAL COMPOSITION — building complex behavior by combining simple
   pure functions, where the output of one becomes the input of the next.

     const compose = (...fns) => x => fns.reduceRight((acc, fn) => fn(acc), x);
     const pipe    = (...fns) => x => fns.reduce((acc, fn) => fn(acc), x);
     // compose applies right-to-left (like math f(g(x))); pipe applies left-to-right

     const trim  = s => s.trim();
     const lower = s => s.toLowerCase();
     const exclaim = s => s + "!";

     const shout = pipe(trim, lower, exclaim);
     shout("  Hello World  "); // "hello world!"
     // Equivalent with compose (reversed order): compose(exclaim, lower, trim)

   IMMUTABILITY — never mutating data in place; instead, creating new copies
   with the desired changes.

     // Mutating (avoid in FP style)
     function addItem(cart, item) { cart.push(item); return cart; }

     // Immutable
     function addItemImmutable(cart, item) { return [...cart, item]; }

     const original = { name: "Alice", age: 30 };
     const updated = { ...original, age: 31 }; // original is untouched

   Why these matter together:
   • Immutability + pure functions make composed pipelines predictable — no
     function in the chain can secretly corrupt shared state that a later
     function depends on.
   • Enables safe memoization, easy undo/redo (keep old references), simpler
     debugging (no "who mutated this?" hunting), and efficient change
     detection in UI frameworks (React/Redux compare references instead of
     deep-diffing, assuming immutability is respected — `prevState !==
     newState` is a cheap way to detect "something changed").
   • Libraries like Immutable.js, Immer, and Object.freeze (shallow) support
     enforcing/simplifying this style in JS, which has no built-in
     structural immutability for objects/arrays otherwise.

────────────────────────────────────────────────────────────────────────────────
Q72: What are common JavaScript security concerns? (XSS, eval, prototype
     pollution)
──────────────────────────────────────────────────────────────────────────────
A: XSS (CROSS-SITE SCRIPTING) — occurs when untrusted user input is inserted
   into the DOM as executable HTML/JS, allowing an attacker to run arbitrary
   script in another user's browser session (steal cookies/tokens, deface
   the page, perform actions as the victim).

     // ⚠️ VULNERABLE — user input rendered as HTML
     el.innerHTML = userSuppliedComment;  // if comment contains
                                            // "<img src=x onerror=stealCookies()>"
                                            // it EXECUTES

     // ✅ SAFE — treats input as plain text, never parsed as HTML
     el.textContent = userSuppliedComment;

   Mitigations:
   • Prefer textContent over innerHTML for untrusted content.
   • Sanitize any HTML that must be inserted (e.g., DOMPurify library).
   • Use a Content-Security-Policy (CSP) header to restrict script sources
     and disallow inline scripts.
   • Store sensitive tokens in HttpOnly cookies (inaccessible to JS/XSS)
     rather than localStorage.
   • Frameworks like React auto-escape interpolated content by default
     (dangerouslySetInnerHTML is an explicit, named opt-out for a reason).

   AVOID eval() and the Function() constructor with any untrusted input —
   both execute arbitrary strings as JavaScript code, a direct code-
   injection vector:
     eval(userInput); // ⚠️ never do this with untrusted input
   Also avoid setTimeout("codeString", delay) / setInterval("codeString",
   delay) — passing a STRING (rather than a function reference) to these
   internally uses eval-like behavior.
   `eval` also disables certain JS engine optimizations even when unused
   input is safe, since the engine must conservatively assume any variable
   could be accessed/modified through it.

   PROTOTYPE POLLUTION — an attacker-controlled object merge/deep-clone
   (e.g., a naive recursive Object.assign-like merge of user JSON input)
   that includes a "__proto__" key, allowing the attacker to inject
   properties onto Object.prototype itself — affecting EVERY object in the
   application:
     const malicious = JSON.parse('{"__proto__": {"isAdmin": true}}');
     deepMerge({}, malicious);
     ({}).isAdmin; // true ⚠️ — polluted globally, affects all plain objects

   Mitigations: validate/sanitize keys during merges (reject "__proto__",
   "constructor", "prototype"), use Object.create(null) or Map for
   user-controlled key/value data instead of plain objects, use
   Object.freeze(Object.prototype) defensively, keep dependencies (lodash,
   etc.) updated (several had CVEs for exactly this).

   OTHER — avoid exposing sensitive data in client-side JS (API keys
   belong server-side); validate/sanitize on the SERVER too (client-side
   validation is a UX nicety, never a security boundary, since it can be
   bypassed entirely); be cautious with JSON.parse on untrusted data sources
   combined with custom revivers.
